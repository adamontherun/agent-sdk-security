Chapter 19 · Part V — Production Deployment

Deploying on AWS Lambda MicroVMs

Chapter 18 ran the agent on ECS, where you typically pack many sessions into one long-running task and they share that task's kernel. This chapter runs it on AWS Lambda MicroVMs, a service that launched in June 2026 and gives every session its own Firecracker microVM, so no two sessions share a kernel. That is the isolation from chapter 12 turned into a managed API. Because the service is new, this chapter grounds each claim in the current CLI and docs rather than memory, and it is honest about one gap: Terraform has no native resource for the microVM itself yet, so the lifecycle runs through the aws lambda-microvms command set, which this machine's AWS CLI (2.35.15) already ships.

⏭ Still evolving

Lambda MicroVMs is weeks old at the time of writing. The specification numbers, region list, CLI flags, and the whole build-run-connect-terminate lifecycle in this chapter were verified by running them against a real account during drafting, not just read from the docs. That live run also settled the IAM trust principal: the build role is assumed by lambda.amazonaws.com with sts:AssumeRole and sts:TagSession, which the build accepted. A service this young can still move, so treat the exact limits as a snapshot and re-run aws lambda-microvms <cmd> help before you build against a specific flag.

Image, then launch

Regular Lambda hands you a function and cold-starts it. MicroVMs split the work into a build step and a launch step. You package your application and its Dockerfile, upload the artifact to S3, and call create-microvm-image with the S3 URI, a Lambda-managed base image ARN, and a build role. What AWS does next is the interesting part: "Lambda runs your Dockerfile, initializes your application, and takes a Firecracker snapshot of the running environment's memory and disk state." Every launch from that image then resumes from the snapshot instead of booting cold.

The --code-artifact argument is a tagged union whose only key is uri, and the build is asynchronous: the image moves from CREATING to CREATED, and you poll get-microvm-image until it lands. Those are details from the shipped CLI on this machine, not the announcement blog, which is the point of checking the CLI before writing.

Per-session isolation is the whole pitch

The reason to reach for MicroVMs over a Fargate task is the isolation boundary. AWS states it plainly: "Each session runs in its own dedicated MicroVM with no shared kernel and no shared resources between users, so untrusted code supplied by one user is contained to their execution environment." This is the Firecracker guarantee from chapter 12, now something you get from an API call instead of a jailer you operate yourself.

That sentence is the deciding factor for one specific kind of agent: the one that lets end users submit code for the agent to run. Fargate already runs each task in its own Firecracker microVM, so the task is isolated from the host. The gap is granularity: you normally pack many user sessions into one long-running task, and those sessions share that task's kernel and its containers. A guest-kernel exploit or a container escape between sessions has somewhere to go. With a microVM per session, a user's untrusted code sees a kernel that is theirs alone, and breaking out means breaking the hypervisor, not a namespace shared with another tenant.

The lifecycle: run, suspend, resume

run-microvm launches a microVM from an image; it starts in PENDING and moves to RUNNING. Reachability comes from the two network-connector flags, not from a security group: the Lambda-managed ALL_INGRESS connector lets HTTPS in, and INTERNET_EGRESS lets the app reach out. Two more flags carry the cost and safety controls. An --idle-policy auto-suspends the microVM after an idle window and can auto-resume it on the next request, and --maximum-duration-in-seconds caps the total runtime. The CLI gives the cap a precise bound: valid range 1 to 28,800 seconds, which is the 8-hour platform maximum. Suspend preserves full memory and disk state; resume restores it.

The idle policy maps well onto the hybrid session pattern from chapter 17. An agent that sits idle between user check-ins suspends, stops billing for compute, and resumes with its transcript and process state intact when the next request arrives. You get the pause-and-resume behavior without wiring up a SessionStore, because the microVM snapshot is the state.

Reaching the microVM: short-lived tokens

A running microVM has a public HTTPS endpoint, but there is no unauthenticated way in. To talk to it you mint a short-lived token with create-microvm-auth-token, scoped to specific ports and expiring in at most 60 minutes, then attach it to a plain HTTPS request in the X-aws-proxy-auth header. The token, not a security group, is what gates access, which suits a per-session model where sessions come and go faster than firewall rules would. A request with no token is rejected outright.

The whole lifecycle above was run end to end against a real account while writing this chapter, from a Python app packaged exactly as the getting-started guide describes. The running microVM reported its own kernel, which is the isolation claim made concrete: an Amazon Linux 2023 ARM64 guest kernel that belongs to that microVM alone, with the app as PID 1.

🏭 In production

The same endpoint with no token answers 403. Every microVM gets a public HTTPS URL, so the token is the entire access control: there is no unauthenticated path in, and no security group to misconfigure. Mint tokens short-lived and per session, and treat a leaked token like a leaked credential, because for the 60 minutes it lives, it is one.

Terraform has no resource for this yet

You would reasonably want to declare a microVM image the way chapter 18 declared an ECS task. You cannot, at least not yet. The AWS provider that terraform init installed for this course (v5.100.0) exposes 1,526 resources, and none of them match microvm:

That is expected for a service this new, and pretending otherwise would be the mistake. So the split is deliberate: Terraform manages the surrounding pieces it does have resources for (the S3 bucket for the code artifact, the build and execution IAM roles), and the microVM image and lifecycle run through the CLI in examples/ch19/scripts/lifecycle.sh. The Terraform in examples/ch19/terraform/ validates the same way chapter 18's does.

⚠ Gotcha

The build role's trust policy is easy to get subtly wrong. The getting-started guide shows lambda.amazonaws.com assuming it with both sts:AssumeRole and sts:TagSession, and a build fails if you omit sts:TagSession. That is what the Terraform here uses, and what the live build in this chapter ran against. Once you know your account and image ARNs, tighten it further with an aws:SourceArn condition so only your own images can assume the role.

When MicroVMs beat ECS for this course's threat model

Both targets can run the same hardened agent behind the same egress proxy. The deciding question is what the sessions contain. If your agent processes your own content and the tenants are your own application's users, chapter 18's ECS deployment is simpler to operate and the shared kernel is an acceptable risk. If your agent runs code that end users supply, so one session is genuinely hostile to another, the per-session microVM boundary is worth the extra moving parts. AWS positions the service for exactly that case, and the sentence about no shared kernel between users is the reason. A shell-in-a-box product, a data-science notebook that executes arbitrary user code, a coding agent that runs whatever a customer pastes: these are the workloads where a container namespace is not the boundary you want, and a microVM is.
