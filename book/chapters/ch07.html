Chapter 7 · Part II — Permissions & Settings

Hooks as a Security Control

Chapter 6 ended on a boundary permission rules can't draw: they can gate a command, but not the URL inside it. A hook can. A PreToolUse hook sees the full tool call before any rule is consulted, runs your code, and returns a decision. That makes it the place to enforce the checks the rule language can't express. This chapter is the one you can do for real without a logged-in model, because a hook is just a program that reads JSON on stdin and writes a decision to stdout.

Where hooks sit in the pipeline

In the SDK's six steps, hooks are step 1. They run before deny rules, before ask rules, before the permission mode, before everything. That position is the source of both their power and their limit. A hook can block a call that every rule would have allowed, which is why the docs suggest putting checks that must run on every tool call into a PreToolUse hook rather than canUseTool. But a hook can't force a call through against a deny rule. The docs are explicit that hook decisions don't bypass permission rules: a hook returning allow skips the interactive prompt, but deny and ask rules are still evaluated, so a managed deny always wins.

The asymmetry is exactly what you want from a security layer. A hook can only ever tighten the decision, never loosen it past the deny wall from Chapter 4. You can hand an agent a broad Bash allow and use a hook to carve dangerous commands back out, confident the hook can't accidentally re-enable something a deny rule forbade.

The stdin/stdout contract

A Claude Code hook is an executable. When a tool call fires, Claude Code runs your command and passes the event as JSON on stdin. For a Bash PreToolUse event, that JSON looks like this:

Your script reads that, decides, and communicates back through the exit code and stdout. The two channels mean different things. The exit codes are the blunt instrument: exit 0 means no objection and the normal permission flow proceeds, exit 2 blocks the call and sends stderr back to Claude as feedback, and any other code lets the call proceed after logging a hook error. For structured control you exit 0 and print a JSON object to stdout instead. Don't mix the two: Claude Code ignores the JSON if you exit 2.

The structured form is a hookSpecificOutput object carrying a permissionDecision of "allow", "deny", or "ask", plus a permissionDecisionReason that Claude sees. Critically, "allow" skips the interactive prompt but does not override permission rules: a matching deny still blocks, a matching ask still prompts. Which is the pipeline behaviour from Chapter 4, now visible from the hook's side.

A real hook: enforce a URL allowlist

Chapter 6 showed that a Bash(curl ...) rule can't reliably constrain a URL. The hook below can, because it parses every http(s) URL out of the command string and checks each host against an allowlist. It also blocks edits to a few protected files. It emits the structured deny form on a block and stays silent otherwise, so an allowed call falls through to the normal permission flow.

The host check is the part worth reading closely. It allows a host only when it equals an allowed domain or is a subdomain of one, so api.github.com passes but github.com.evil.com does not, because the latter ends in .evil.com, not .github.com. That's the same lookalike defense WebFetch's dot-crossing rule gives you, written by hand for the Bash case.

🏭 In production

Write security hooks to fail closed. If url_guard.py receives malformed JSON, it denies rather than letting the call through, because a hook that crashes on bad input and exits nonzero would let the call proceed (any exit code other than 0 or 2 lets the action run). The same discipline applies to the SDK's in-process hooks: catch your own exceptions, since an unhandled one can interrupt the agent. A guard that errors open is worse than no guard, because you'll trust it.

When several hooks fire on one event they run in parallel, and the most restrictive result wins: the docs order it deny over defer over ask over allow, so any single deny blocks the call regardless of what the others return. Write each hook to stand on its own rather than assuming another ran first; completion order is not deterministic.

🔬 Under the hood

A hook's allow and a permission mode's auto-approval both skip the prompt, but only the mode approval can shadow canUseTool. A hook runs at step 1 and never removes step 6 from the pipeline; a bare allow rule or bypass mode does. That's why the shadowing fix from Chapter 4 is to move the check into a hook: the hook is guaranteed to run on every call, and its deny survives every mode including bypassPermissions.
