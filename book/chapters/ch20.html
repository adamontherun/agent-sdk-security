Chapter 20 · Part V — Production Deployment

Observability and Audit

Every control in this book can be running correctly and you would not know it from the outside. The permission pipeline denies a call, the sandbox blocks a write, the proxy refuses a domain, and none of it is visible unless something records it. This closing chapter is about that record: the telemetry the Agent SDK exports, the proxy logs that show what the agent actually reached over the network, and the first moves when you suspect a prompt injection drove an action. It answers the question the previous nineteen chapters set up but never closed: how do you know it is working?

Telemetry from the SDK

The SDK does not produce telemetry itself. It passes OpenTelemetry configuration through to the claude subprocess, which has the instrumentation built in and exports directly to your collector. So you configure it with environment variables at the container or orchestrator level, and every query() call picks them up with no code change. Telemetry stays off until you set CLAUDE_CODE_ENABLE_TELEMETRY=1 and choose at least one exporter, per the observability documentation.

There are three independent signals, each with its own switch. Metrics carry counters for tokens, cost, sessions, and tool decisions. Log events carry a structured record for each prompt, API request, API error, and tool result. Traces carry spans for each interaction, model request, tool call, and hook, and they are in beta, so they need one extra flag.

SignalEnable with
MetricsOTEL_METRICS_EXPORTER
Log eventsOTEL_LOGS_EXPORTER
Traces (beta)OTEL_TRACES_EXPORTER plus CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1

For a multi-tenant deployment, attach the end user's identity as resource attributes on each call so tool activity is attributable to the user the agent acted for, not just to your service's credential. Percent-encode the values, because OTEL_RESOURCE_ATTRIBUTES reserves commas, spaces, and equals signs. With end-user identity attached, the tool_decision, tool_result, and permission_mode_changed events become a per-user audit trail you can forward to a SIEM.

⚠ Gotcha

Do not set console as an exporter value when running through the SDK. The console exporter writes telemetry to standard output, which is the same channel the SDK uses as its message pipe, so it corrupts the stream. To inspect telemetry locally, point OTEL_EXPORTER_OTLP_ENDPOINT at a local collector or an all-in-one Jaeger container instead.

Source: Observability with OpenTelemetry

🏭 In production

Telemetry is structural by default: durations, model names, tool names, and token counts, but not the content the agent read or wrote. Prompt text and tool inputs stay out of exports unless you opt in with OTEL_LOG_USER_PROMPTS=1, OTEL_LOG_TOOL_DETAILS=1, or OTEL_LOG_TOOL_CONTENT=1. Leave these unset unless your observability pipeline is approved to store the data your agent handles. Turning them on sends the file paths, shell commands, and tool bodies your agent touched to your collector, which is a data-handling decision, not a logging convenience.

The proxy log is the network truth

SDK telemetry tells you what the agent decided and which tools ran. It does not, by itself, tell you what left the box. That is the proxy's job. The Squid access log from chapter 15, and the sidecar logs from chapter 18, are the record of every destination the agent actually reached over the network. When the two disagree, the proxy wins, because it sits at the boundary the agent cannot see past.

This is why the whole course routes egress through a proxy rather than trusting the agent to behave. A prompt injection can talk the model into trying to reach an attacker's server. It cannot make the request succeed if the proxy's allowlist does not include that domain, and the attempt lands in the access log either way. Configure the proxy to log in a format you can parse, keep those logs somewhere the agent's container cannot write to, and you have an audit trail that survives a compromised session.

Rotating the credential the agent never sees

Part IV's proxy pattern keeps the real credential outside the agent's boundary: the agent makes a request without it, and the proxy injects it before forwarding. That design also makes rotation cheap and safe. Because the key lives in one place (Secrets Manager in chapter 18, resolved into the sidecar at task start) rather than distributed to every agent, you rotate it by updating the secret and recycling the proxy, and no agent container needs to change. Rotate on a schedule, and rotate immediately if a proxy log shows a credentialed request going somewhere it should not have. The agent never held the old key, so the blast radius of a rotation is the proxy, not the fleet.

When you suspect an injection: first moves

Suppose a user reports the agent did something it should not have, or an alert fires. Three places hold the answer, and they line up with three parts of this book.

Start at the proxy access log, because it shows where the agent tried to go. An unexpected domain, especially a paste site, a raw IP, or the cloud metadata endpoint at 169.254.169.254, is the signature of an exfiltration attempt. Next, read the permission-denial events from the SDK telemetry: a burst of denied tool calls says the model was pushed toward actions your rules blocked, which is what a live injection looks like from the permission pipeline's side. Then check the hook rejections from chapter 7, since a PreToolUse hook firing is the layer catching what the static rules could not express. The triage example turns the first of these into a tool.

Those three lines are what a first-pass triage surfaces from a sample Squid log: two attempts to reach exfiltration destinations and one attempt on the instance-metadata endpoint, which is a classic move to steal cloud credentials. The allowlisted traffic (the Anthropic API, the npm registry, GitHub) produces no finding, so what remains is the short list worth a human's attention.

🔬 Under the hood

The triage tool reads Squid's native access-log format, where each line is space-separated and the request target is the sixth field. For a CONNECT tunnel that target is host:port; for a plain method it is a full URL. Deriving the bare hostname from both shapes and comparing against an allowlist is the entire logic. It is deliberately small, because incident triage tools that nobody can read in a hurry do not get used in a hurry.

The whole stack, in one sentence per layer

This is where the parts meet. The permission pipeline from Part II decides whether a tool call is allowed. The isolation from Part III decides what a call can touch if the process is compromised anyway. The network boundary from Part IV decides where the agent can reach and keeps the credential out of its hands. The hosting model and deployments from Part V decide where all of that runs and how tenants stay apart. Observability is the layer that makes the other four legible: the telemetry, the proxy logs, and the audit trail are how a control you configured becomes a control you can prove is holding. An agent's behavior cannot be fully predicted from its code, which is where this course started. What you can do is bound what the behavior is able to reach, and then watch the boundary.
