Chapter 15 · Part IV — Credentials & the Network Boundary

Squid, Hands-On

This is the chapter where the proxy pattern stops being a diagram. You start from a Squid config that allows everything and, in three steps, turn it into an egress proxy that permits only the hosts you name, decrypts the TLS it is allowed to inspect, and injects a credential the client never sent. Every result below is captured from the real container running on the lab network in this repo. Nothing here is asserted without the command and its output.

The lab, and the image it needs

The repo ships a compose.yaml with two services on a network called lab: the Squid proxy, and a small mock-api (a stock kennethreitz/httpbin) that echoes back the request it received, which is how we will prove injection later. The scaffold used the stock ubuntu/squid image, and it round-trips fine as a plain tunnel. It cannot do the interception this chapter needs, and the reason is worth knowing before you spend an hour debugging it.

TLS bumping requires Squid to mint a fresh certificate for each host it intercepts, which it does through an external helper called security_file_certgen. That helper is only built when Squid is compiled against OpenSSL. The ubuntu/squid image is built against GnuTLS and ships no such helper, so it can peek at a hostname but cannot decrypt. Alpine's squid package is built with OpenSSL and includes the helper. Both facts are checkable inside the image rather than taken on faith:

So the lab builds Squid from a small infra/squid/Dockerfile based on Alpine. The entrypoint.sh initialises the certificate database once with security_file_certgen -c, clears any stale PID file so docker compose restart works, validates the config with squid -k parse, and runs Squid in the foreground. Before the first run you mint the proxy's certificate authority:

That key is the whole game. Anything that trusts it will trust every certificate Squid forges. It stays under infra/squid/certs/, which the repo's .gitignore already excludes, and it never crosses to the agent's side of the boundary.

Step 1 — a domain allowlist

The minimum useful control is an allowlist: the agent reaches the hosts you name and nothing else, with no certificate machinery at all. Squid's dstdomain ACL matches the request's destination host, and for an HTTPS request that host is visible in the CONNECT line and the TLS SNI even without decryption. A leading dot means the domain and all its subdomains.

Copy that variant to infra/squid/squid.conf, bring the lab up, and send two requests from a throwaway curl container on the lab network: one to an allowed host, one to a host that is not on the list.

Squid's own access log records the decision on each side. Note what it can and cannot see here: the host, yes; the path, no. An allowed request is a TCP_TUNNEL, a blocked one is TCP_DENIED, and in both cases the log line stops at host:443 because the traffic inside the tunnel is still encrypted.

Step 2 — SSL bump, so the proxy can see inside

To inspect or rewrite an HTTPS request the proxy has to terminate the TLS itself: present the client a certificate for the host it asked for, read the plaintext request, then open its own connection to the real server. Squid calls this bumping. Modern Squid does it with a step-based model called peek-and-splice, which the wiki introduced to replace the older server-first/client-first actions that it describes as not working well in many cases.

The model runs in up to three steps. At step 1 Squid has only the TCP CONNECT; the client's TLS ClientHello, with the SNI inside it, comes next. Peeking at step 1 reads that ClientHello, so the SNI is available at step 2, where the bump decision is made. The at_step SslBump1 ACL matches that first step. The config peeks at step 1 to pull the hostname forward, then decides: bump anything on the allowlist, and terminate anything else. The http_port gains the ssl-bump flag and the CA to sign forged certificates with, and sslcrtd_program points at the helper that generates and caches them.

The tls-cert=/tls-key= spelling is the current one; older guides show cert=/key=. Rather than trust either, the entrypoint runs squid -k parse on every start, which accepts the directive above without complaint on Squid 6.9. After a rebuild the log confirms the CA is loaded and the port is listening:

Now repeat the request, but tell curl to trust the proxy CA (in a real deployment this certificate goes into the agent container's trust store; here --cacert is the equivalent). Two things prove the bump happened. The certificate the client received for httpbin.org was issued by our CA, not by a public authority. And the access log now shows the full decrypted request line, path and all, where step 1 saw only CONNECT host:443.

⚠ This is a man-in-the-middle. Know the law.

Bumping is not a metaphor. Squid's own wiki states that decrypting HTTPS tunnels constitutes a man-in-the-middle attack from the overall network security point of view, and warns that decrypting HTTPS tunnels without user consent or knowledge may violate ethical norms and may be illegal in your jurisdiction. Squid's decryption features are designed for deployment with user consent or, at the very least, in environments where decryption without consent is legal. Intercepting your own lab traffic on your own machine is fine. Pointing this at traffic you do not own, or that carries other people's data, is a legal and ethical decision, not a technical one, and it is yours to make deliberately. HTTPS was designed to give users an expectation of privacy; bumping removes it.

Source: Squid wiki — Feature: Squid-in-the-middle SSL Bump

Step 3 — inject a credential the agent never held

Because the proxy now decrypts the request, it can also add to it. request_header_add attaches a header to every request bound for an allowlisted host. The agent's own request carries no credential; the upstream still receives one. In production the value comes from a secret store and never appears in the file; it is inline here so the lab is self-contained.

The proof is the echo service. The client sends no Authorization header of its own. The upstream reports having received one. This works on the plain-HTTP mock and on the bumped HTTPS request equally, because request_header_add only touches requests Squid can actually read, which after step 2 includes the decrypted HTTPS. It would do nothing to a connection that was merely tunnelled.

🏭 In production

Once bumping is on, every client must trust the proxy CA to reach even an allowed host, because Squid presents its own certificate for everything. That is a feature: a container whose trust store you did not provision cannot silently reach out past the proxy. It is also the most common reason a freshly wired agent throws certificate verify failed. Install the CA in the agent image's trust store, not on the fly. Chapter 18 wires this exact Squid, CA and all, into an ECS task where the agent container has no route to the internet except through it.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Docker and the compose lab are ready to go.

From the repo root: sh infra/squid/gen-ca.sh once, then docker compose up -d --build, then sh examples/ch15/prove.sh. That script runs all four proofs above against the live container, the allowlist allow and deny, the bump, and both injection paths, so you can watch them pass on your own machine. The labeled squid.conf.step* files under examples/ch15/ are the snapshots at each stage; infra/squid/squid.conf holds the final one.

The challenge for this part lives in chapter 16. If you want to go further with Squid here, try tightening ssl_bump so a host on the allowlist is spliced (allowed through without decryption) while the rest are bumped, and confirm from the access log which requests show a full URL and which stop at CONNECT.
