Chapter 9 · Part III — Isolation Technologies

The Built-In Sandbox

Before you reach for Docker, gVisor, or a microVM, Claude Code already ships an isolation boundary you can turn on with one setting. It uses the operating system's own sandboxing primitives to confine every Bash command and its children to a set of files and network domains you choose. This chapter is about what that boundary actually enforces, the one default that surprises people, and the point past which you need the heavier machinery the rest of Part III builds.

What the sandbox is

The sandbox wraps the Bash tool. When it is on, a command Claude runs and every process that command spawns execute inside an OS-enforced jail that limits which files they can touch and which hosts they can reach. The sandboxing documentation is precise about the mechanism: on macOS it uses the built-in Seatbelt framework, and on Linux and WSL2 it uses bubblewrap for filesystem isolation plus socat to relay network traffic through a proxy. Native Windows is not supported; on Windows you run Claude Code inside a WSL2 distribution. Because the restriction lives in the kernel and not in Claude Code, it holds regardless of what the model decided to run, and it holds even when an allowed command does more than its name suggests.

That last point is the reason the sandbox matters as a security control and not just a convenience. Permission rules decide whether a command runs based on the command string. The sandbox decides what the running process can access once it is already executing. A command that passes your allow rules and then tries to read a file it was never meant to touch is stopped by the kernel, not by a judgement about the command text.

🔬 Under the hood

The same primitives are packaged separately as the @anthropic-ai/sandbox-runtime package, which wraps an entire process rather than just the Bash tool. Subagents run in the same process as their parent and inherit the same sandbox configuration, so enabling the sandbox in a parent session sandboxes the Bash commands its subagents run too.

Filesystem isolation, and the read that is broad by default

Writes are tight. By default a sandboxed command can write only to the current working directory and its subdirectories, plus the session temp directory that $TMPDIR points at. It cannot modify ~/.bashrc, system binaries under /bin, or anything else outside those two locations without explicit permission. When a tool genuinely needs to write elsewhere, you grant that path with sandbox.filesystem.allowWrite rather than dropping the tool out of the sandbox entirely.

Reads are the opposite, and this is the default worth pausing on. The sandbox allows read access to the entire computer except for directories you explicitly deny. The documentation states the consequence plainly: the default still allows reading credential files such as ~/.aws/credentials and ~/.ssh/. A command confined to writing inside your project can still read your cloud keys. If that command also has a network domain it is allowed to reach, you have the two halves of an exfiltration path.

The fix is sandbox.credentials, a settings block whose job is to name the files and environment variables to keep away from sandboxed commands. File entries take "mode": "deny", which blocks reads the same way filesystem.denyRead does. Environment-variable entries with deny are unset before each sandboxed command runs.

There is a second mode for environment variables, mask, that solves a real tension: deny protects a token by deleting it, which also breaks the tool that needed it, like gh or npm. With mask the sandboxed command sees a per-session sentinel value instead of the real credential, and when a request leaves the sandbox for one of the credential's injectHosts, the proxy swaps the sentinel for the real value on the way out. The command never holds the secret, but its requests still authenticate. Masking has a prerequisite the docs are emphatic about: the proxy has to see request contents to substitute the credential, so you must set network.tlsTerminate so the proxy terminates HTTPS itself. Without it, masking fails closed and the sentinel reaches the server unchanged, which Claude Code reports at startup and in /doctor.

⚠ Gotcha

There is no built-in credential deny list. Only the files and variables you name are protected. A fresh sandbox with sandbox.enabled set to true and no credentials block still lets a command read ~/.aws/credentials. Treat the credentials block as required configuration, not an optional extra, and consider setting CLAUDE_CODE_SUBPROCESS_ENV_SCRUB to strip provider credentials from every subprocess regardless of the sandbox.

Network isolation through the built-in proxy

Network access runs through a proxy that lives outside the sandbox, and the starting position is closed: no domains are pre-allowed. The first time a command needs a new domain, Claude Code prompts for approval, and as of v2.1.191 approving a host allows it for the rest of the session so repeat connections do not prompt again. You skip the prompt entirely by listing hosts in allowedDomains. For a fleet, managed settings can set allowManagedDomainsOnly, which blocks non-allowed domains outright instead of prompting and honors only the allowedDomains that come from managed settings.

The proxy makes its decision from the hostname the client asks for. By default it does not terminate or inspect TLS. That keeps it simple and fast, and it is the source of the sharpest limitation in this whole chapter, which the next section gets to.

Two ways to approve sandboxed commands

The /sandbox panel offers two modes, and they are about approval flow, not about the strength of the boundary. In both modes the sandbox enforces the same filesystem and network restrictions. In auto-allow mode, sandboxed commands run without a permission prompt because the sandbox boundary contains them; commands that cannot be sandboxed fall back to the regular permission flow. In regular-permissions mode, every Bash command still goes through the normal prompts even when it is sandboxed, which trades convenience for more explicit control.

Even in auto-allow mode, some things still stop a command. Explicit deny rules always apply. An rm or rmdir aimed at /, your home directory, or other critical paths still prompts. Content-scoped ask rules like Bash(git push *) still force a prompt. Auto-allow is a statement that the sandbox is a sufficient boundary for ordinary commands, not a blanket approval of everything.

🏭 In production

For a managed deployment, the three keys that turn the sandbox from a convenience into a gate are enabled, failIfUnavailable, and allowUnsandboxedCommands. The first turns it on. The second makes a missing dependency such as bubblewrap a hard startup failure rather than a silent fall-back to unsandboxed execution. The third disables the dangerouslyDisableSandbox escape hatch, so a command that fails under the sandbox cannot be retried outside it. Deliver them through managed settings and Claude Code uses the managed values for these boolean keys and ignores anything a developer sets locally.

Where the guarantees end

The documentation has a security-limitations section, and reading it is what separates using the sandbox well from trusting it too far. The most important limitation follows directly from the proxy design. Because the proxy allows or blocks based on the client-supplied hostname without inspecting TLS, code inside the sandbox can potentially use domain fronting or similar techniques to reach a host outside the allowlist. Allowing a broad domain like github.com widens this considerably. If your threat model needs a hard network boundary, you configure a custom proxy that terminates TLS and inspects traffic, which is exactly what Part IV builds with Squid.

A few other limitations are worth holding in mind. Allowing a Unix socket through allowUnixSockets can hand back the access the sandbox took away; the docs call out /var/run/docker.sock specifically, since access to the Docker socket effectively grants access to the host. Overly broad write permissions can enable privilege escalation, for instance by letting a command write to a directory on $PATH or to a shell startup file. And two weakening options exist for compatibility that reduce the guarantee when set: enableWeakerNestedSandbox, which lets the sandbox run inside an unprivileged container by bind-mounting the container's /proc instead of a fresh one, and enableWeakerNetworkIsolation, used when routing through a MITM proxy with a custom CA on macOS. Both should be reserved for cases where an outer boundary already provides the isolation you need.

🔬 Under the hood

The honest framing is that the sandbox is real OS-level enforcement on a shared kernel. Sandboxed processes use the same kernel as the host, so a kernel vulnerability could in theory allow escape. For a developer laptop or many CI jobs that boundary is a large improvement over nothing, and it costs one settings block. When the threat model includes untrusted content or multiple tenants, you want a boundary that does not share the host kernel. That is the gap chapters 10 through 12 fill, in increasing order of strength and cost.

Effective sandboxing needs both halves at once. Without network isolation a compromised command can exfiltrate a file it was allowed to read; without filesystem isolation it can backdoor a resource to gain network access. When you widen a default on one side, check that you have not undone a restriction on the other. That habit is the through-line for the rest of this part: every stronger boundary we build still asks the same question about what the code inside can read and where it can send what it reads.
