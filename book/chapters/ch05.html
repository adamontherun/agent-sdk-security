Chapter 5 · Part II — Permissions & Settings

Permission Modes Compared

A permission mode is a single setting that changes what step 4 of the pipeline does to every unresolved tool call. The names sound self-explanatory, which is the danger: acceptEdits accepts more than edits, bypassPermissions doesn't bypass everything, and both propagate to subagents in ways that surprise people. This chapter pins down what each mode auto-approves to the exact command, and answers the question that actually matters for a production deployment: when is bypassPermissions safe?

The six modes at a glance

Claude Code and the SDK share the same set of modes. The table below is the short version; the sections after it hold the details that bite.

ModeWhat it auto-approves at step 4
defaultNothing. Each tool prompts on first use (labeled Manual in recent CLI builds).
acceptEditsFile edits and a fixed list of filesystem Bash commands, scoped to the working directory and additional directories.
planRead-only exploration. Writes are never auto-approved; they route to your callback.
autoTool calls a model classifier judges to match your request. A research preview.
dontAskNothing new. It turns every would-be prompt into a denial instead.
bypassPermissionsEverything that reaches step 4, except calls caught by explicit ask rules or the circuit breaker.

The descriptions come from the Claude Code permission modes table and the SDK mode table, which agree on behaviour. In Python you set the mode with ClaudeAgentOptions(permission_mode="acceptEdits"), or change it mid-session with client.set_permission_mode(...).

What acceptEdits actually approves

acceptEdits is the mode people reach for to stop babysitting a refactor, and it auto-approves more than the Edit and Write tools. It also approves a fixed list of filesystem Bash commands. The SDK docs name them: mkdir, touch, rm, rmdir, mv, cp, and sed. Both the edits and the commands apply only to paths inside the working directory or additionalDirectories; paths outside that scope, and writes to protected paths, still prompt.

That rm is on the list surprises people who read the mode name as "accept edits." Running examples/ch05/acceptedits_demo.py makes the scope concrete. The model is configured with a working directory of /home/dev/project and one additional directory, /home/dev/shared:

Read down the verdicts. An edit inside the project is approved; the same edit to /etc/hosts is not. An rm of a project file is approved; an rm of /etc/passwd is not, because it's out of scope. And rm -rf / prompts even though rm is on the approved list, which is the circuit breaker we turn to next. A non-filesystem command like curl was never a candidate for acceptEdits and prompts as usual.

⚠ Gotcha

acceptEdits scopes on the working directory, so anything you add to additionalDirectories widens the auto-approval blast radius for rm, mv, and the rest, not just for reads. If you extend an agent's directories for convenience, you have also extended where it can delete files without asking. Keep that list as short as the task allows.

Source: Agent SDK permissions — acceptEdits mode

The bypassPermissions circuit breaker

bypassPermissions skips permission prompts for everything that reaches step 4. The name suggests a clean off switch, but two categories of call still stop it. Explicit ask rules still force a prompt, and a small set of catastrophic removals still prompt as a circuit breaker against model error. The warning in the docs spells out that even in bypass mode, writes to .git, .config/git, .claude, .vscode, .idea, .husky, .cargo, .devcontainer, .yarn, and .mvn are skipped without a prompt, but removals targeting the filesystem root or home directory, such as rm -rf / and rm -rf ~, still prompt.

Read that carefully, because it cuts against intuition. The dotfile directories in that list are not protected in bypass mode; the warning is telling you the mode will happily write to your .git directory. Only the root and home removals are held back, and only as a last-ditch guard against the model doing something ruinous, not as a security boundary you should design around.

Modes propagate to subagents

A permission mode is not confined to the agent you set it on. The SDK documents that when the parent uses bypassPermissions, acceptEdits, or auto, all subagents inherit that mode and it cannot be overridden per subagent. That's a sharper edge than it first appears. Subagents can carry different system prompts and behave less conservatively than your main agent, and a subagent spawned from a bypass-mode parent gets full, autonomous system access with no way to walk it back for that child. An explicit ask rule is the one thing that still forces a prompt inside the subagent.

🏭 In production

If your agent uses subagents at all, treat the parent's permission mode as the mode the whole tree runs in. A convenient acceptEdits on the parent becomes acceptEdits on every Explore and Plan subagent underneath it, deleting files in scope without prompting. Design the mode for the least trusted node in the tree, then use deny and ask rules, which subagents can't shed, to hold the line where a mode can't.

When bypassPermissions is actually safe

The honest answer is narrow. bypassPermissions is safe only inside a real isolation boundary, one where the agent having full system access doesn't matter because the system is disposable and contains nothing you'd mind losing. The docs put it the same way: only use this mode in isolated environments like containers or VMs where Claude Code can't cause damage.

The failure is running it on a developer's laptop or a shared host because prompts are annoying, and then leaning on the mode's own guards to save you. Those guards, the ask-rule exception and the root-removal circuit breaker, are not an isolation boundary. They stop a couple of specific mistakes; they don't stop an agent reading your credentials, writing to your .git, or reaching the network. A real boundary is a container the agent can't escape, a microVM, or a network the agent can't see past. Building those boundaries is the subject of Part III, and bypassPermissions belongs strictly inside one of them.

Two adjacent guardrails are worth naming. You can forbid the mode outright by setting permissions.disableBypassPermissionsMode to "disable" (Chapter 8 pins it in managed settings so nobody can flip it back), and a related note from Part III: the standalone --dangerously-skip-permissions flag is blocked when running as root outside a recognized sandbox, precisely because root plus no prompts can rewrite anything on the host.

🔬 Under the hood

A permission mode is not the same thing as the Bash sandbox's auto-allow mode, even though both cut prompts. A mode decides whether a tool call runs at step 4 of the pipeline. The sandbox decides what a Bash command can touch once it runs, and its auto-allow substitutes the OS boundary for the prompt. They compose: you can run default mode with sandbox auto-allow and still get unattended Bash execution, contained by the kernel rather than trusted by the classifier. Part III separates the two properly.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — the Python toolchain, Docker, the AWS CLI, and Terraform are already installed.

Run python3 examples/ch05/acceptedits_demo.py and add a case: an rm of a file in the additional directory. Predict the verdict, then confirm it.

Then open challenges/ch05/challenge.py and implement accept_edits_approves(), the exact command list, the in-scope check, and the rm -rf / circuit breaker.
