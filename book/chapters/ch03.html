Chapter 3 · Part I — Threat Model & Foundations

Security Principles: Boundaries, Least Privilege, Defense in Depth

Everything from here on is a variation on three ideas. Put your secrets on the far side of a boundary the agent can't cross. Give the agent only the access its task actually needs. And assume any single control will fail, so stack several that fail independently. The secure-deployment guide names these as security boundaries, least privilege, and defense in depth. This chapter defines each one concretely, then maps the four remaining parts of the course onto them, so you can see where every later technique fits before you meet it.

Security boundaries

A boundary is a line between things you trust to different degrees. The secure-deployment guide defines it this way: "A security boundary separates components with different trust levels. For high-security deployments, you can place sensitive resources (like credentials) outside the boundary containing the agent. If something goes wrong in the agent's environment, resources outside that boundary remain protected."

The move that makes this real for agents is the proxy pattern, which Part IV builds by hand. Rather than handing the agent an API key, you run a proxy outside its environment that injects the key into outgoing requests. In the guide's words, "the agent can make API calls, but it never sees the credential itself." Line that up against the lethal trifecta from chapter 1. Prompt injection can trick an agent into sending data anywhere it's able to reach, but it cannot make the agent leak a secret the agent was never given. The credential sits outside the boundary, so a compromise inside the boundary can't reach it. That single sentence, a credential the agent never holds is a credential it can never leak, is the reason the network boundary gets its own part of this book.

Least privilege

Least privilege is the discipline of granting only what the task requires and nothing spare. The secure-deployment guide frames it as restricting "the agent to only the capabilities required for its specific task," and gives a compact table of where those restrictions apply and what each one looks like.

Least-privilege restrictions, from the secure-deployment guide
ResourceRestriction options
FilesystemMount only needed directories, prefer read-only
NetworkRestrict to specific endpoints via proxy
CredentialsInject via proxy rather than exposing directly
System capabilitiesDrop Linux capabilities in containers

Four rows, four questions to ask of any deployment. Which directories does the agent truly need, and can they be read-only? Which network endpoints, and can everything else be blocked? Do credentials have to be present in the agent's environment, or can a proxy hold them? Have the Linux capabilities that enable privilege escalation been dropped? The challenge for this chapter turns exactly these four rows into a checker that flags an under-restricted config, and the example runs it against a wide-open default and a hardened one.

The four violations in the first row map one-to-one onto the four table rows, and each is addressed by a specific technique later in the course. A writable root filesystem becomes a read-only mount in chapter 16. Unrestricted network becomes an allowlisting egress proxy in chapter 15. Credentials in the environment move behind the proxy in chapter 14. Undropped capabilities get dropped in the hardened container of chapter 10. The principle is abstract; the fixes are concrete, and you build each one.

Defense in depth

No single control on that list is trustworthy enough to be the only one. Defense in depth is the habit of layering controls that fail independently, so that one being bypassed doesn't open the whole system. The secure-deployment guide's threat-model example is the clearest statement of why: "if an agent processes a malicious file that instructs it to send customer data to an external server, network controls can block that request entirely." The model's own resistance to injection might catch that file. If it doesn't, a network control that never saw the instruction still stops the exfiltration. The two layers don't share a failure mode, which is the whole point.

For deployments that need more than the defaults, the guide lists the controls to layer: container isolation, network restrictions, filesystem controls, and request validation at a proxy. Notice these are the subjects of Parts III and IV. Defense in depth isn't a separate technique you bolt on at the end; it's the reason the course covers isolation and the network boundary as distinct layers rather than picking one and calling it done. As the guide puts it, "the right combination depends on your threat model and operational requirements," which is why chapter 13 is a decision framework rather than a single recommendation.

🔬 Under the hood

These three principles aren't specific to agents. The secure-deployment guide is explicit that "the same principles that apply to running any semi-trusted code apply here: isolation, least privilege, and defense in depth." What's new is the threat model from chapter 1, not the toolbox. Containers, proxies, dropped capabilities, and read-only mounts are decades-old operational security. The shift is that you now point them at a process whose next action is generated from untrusted input, so you apply them by default rather than only to code you've already decided is hostile.

Source: Securely deploying AI agents

⚠ Gotcha

Not every deployment needs every layer, and treating maximum security as the only correct answer is its own mistake. The guide draws the line clearly: "A developer running Claude Code on their laptop has different requirements than a company processing customer data in a multi-tenant environment." Least privilege cuts both ways. Over- restricting a low-risk local workflow buys friction, not safety. The work is matching the controls to the threat model, which means you have to actually name the threat model first, the way chapter 1 named the trifecta legs.

Source: Securely deploying AI agents

How the rest of the course builds on this

With the three principles in hand, the shape of the remaining four parts is easy to see, because each part hardens one layer of the same picture.

Part II hardens the permission layer: the evaluation pipeline a tool call moves through, the permission modes, the tool-specific rules, hooks as a programmable control, and managed settings that let an organization lock a fleet down. This is least privilege applied to what the agent is allowed to attempt in the first place. Part III hardens the isolation boundary: the built-in sandbox, hardened containers, gVisor, and microVMs with Firecracker, ending in a framework for choosing among them. This is defense in depth applied at the process boundary, so that a control the model bypasses runs into a wall the operating system enforces. Part IV hardens credentials and the network: the proxy pattern, a real Squid egress proxy you build, and filesystem configuration. This is the security-boundary principle made concrete, putting secrets and egress on the far side of a line the agent can't cross. Part V puts it into production: hosting the Agent SDK, deploying on Amazon ECS and on AWS Lambda MicroVMs, and the observability to know when something went wrong. Every one of those chapters is one of these three principles, turned into infrastructure you run.
