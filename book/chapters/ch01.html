Chapter 1 · Part I — Threat Model & Foundations

Why Agent Security Is Different

You can read every line of an agent's source and still not know what it will do, because its actions aren't in the source. They're generated at runtime from whatever the model reads: your prompt, the files it opens, the web pages it fetches, the output of the last command it ran. That one fact reshapes the whole security problem. This chapter is about what changes when the thing you're securing writes its own next move, and it names the two ideas the rest of the course keeps coming back to: prompt injection and the lethal trifecta.

Code you can read, behavior you can't predict

A normal program has a fixed set of paths through it. Given the same inputs it does the same thing, and if you want to know whether it can delete a file or open a socket, you can read the code and find out. An agent breaks that assumption. As the Claude Agent SDK secure-deployment guide puts it, "Unlike traditional software that follows predetermined code paths, these tools generate their actions dynamically based on context and goals." The flexibility is the point. It's also the problem.

The tools an agent can reach are still fixed, and this course spends most of its length on constraining exactly those. What isn't fixed is the sequence of calls the model chooses to make, and the arguments it passes. A code reviewer agent that only ever needs to read files can, in principle, be steered into running a shell command, because deciding to run that command is a runtime choice made from context, not a branch you can grep for. Reviewing the agent's code tells you the ceiling on what's possible. It doesn't tell you what will happen on any given run.

🔬 Under the hood

Claude models are trained to resist manipulation, and that resistance is real and load-bearing. The secure-deployment guide is explicit that it isn't a complete defense on its own: "Agents can take unintended actions due to prompt injection (instructions embedded in content they process) or model error," and "Defense in depth is still good practice though." The model's judgment is one layer. The permission rules, sandboxes, and proxies in the rest of this course are the layers that hold when it slips.

Source: Securely deploying AI agents — Threat model

Prompt injection

The name for the core failure mode is prompt injection. The Claude Code security documentation defines it as "a technique where an attacker attempts to override or manipulate an AI assistant's instructions by inserting malicious text." The model reads instructions and content in the same stream of tokens and has no reliable way to tell which side an instruction came from. Simon Willison, who coined the term, describes the root cause plainly: LLMs "are unable to reliably distinguish the importance of instructions based on where they came from." He draws the analogy to SQL injection, where trusted query structure and untrusted user data get concatenated into one string and the database can't tell them apart.

This is not a fringe concern. Prompt injection sits at the top of the OWASP Top 10 for LLM Applications as LLM01, described there as manipulating an LLM via crafted inputs to cause unauthorized access, data breaches, and compromised decision-making. It is a recognized, catalogued class of vulnerability, not a hypothetical.

The concrete version the secure-deployment guide gives is worth holding onto because it's so ordinary: "if a repository's README contains unusual instructions, Claude Code might incorporate those into its actions in ways the operator didn't anticipate." No exploit, no memory corruption. Just text in a file the agent was asked to read, phrased as an instruction, treated as one.

⚠ Gotcha

The untrusted content doesn't have to be visible to a human skimming the file. It can sit in a code comment, a commit message, an HTML element styled to be invisible, the alt text of an image, or a page the agent fetches three links deep from the one you pointed it at. If your mental model is "I trust this repo, so I trust what the agent reads in it," remember that a dependency's README, a pasted stack trace, or an issue comment from a stranger all land in the same context window as your own instructions.

The lethal trifecta

Prompt injection explains how an attacker gets a foothold. It doesn't, by itself, explain when that foothold turns into real damage. Simon Willison's framing for that is the lethal trifecta: three capabilities that are each fine alone but dangerous together.

The first is access to private data, which is one of the most common reasons to give an agent tools in the first place. The second is exposure to untrusted content, meaning any path by which text an attacker controls reaches the model. The third is the ability to communicate externally, which Willison calls exfiltration: some way for the agent to send data back out. His conclusion is the sentence to memorize: "If your agent combines these three features, an attacker can easily trick it into accessing your private data and sending it to that attacker."

The secure-deployment guide gives the canonical example: "if an agent processes a malicious file that instructs it to send customer data to an external server, network controls can block that request entirely." Read that against the three legs. Customer data is the private data. The malicious file is the untrusted content. The request to an external server is the exfiltration path. All three are present, so the attack works. The guide's own fix is to remove one leg: block the outbound request, and the private data has nowhere to go.

Only the first deployment above is marked lethal, and it's the only one with all three legs. The code reviewer holds private data but takes no untrusted input and has no way out. The docs summarizer touches untrusted pages and can send mail, but has no secrets worth stealing. Each of the safe three is one capability short of the trifecta. That's the design lever the whole rest of this course pulls on: you rarely get to remove the private data or the untrusted content, because those are usually the job, so most hardening is about controlling the third leg, the ability to communicate out.

🏭 In production

When you audit a deployment, count the legs before you reach for any specific control. An internal agent that reads a private database and also browses arbitrary URLs a user pastes in already has all three, even if "sending data out" is just an HTTP GET with the secret in the query string. The multi-tenant case is worse: one tenant's untrusted content plus another tenant's private data, sharing one agent, is the trifecta assembled from parts no single tenant could see was dangerous. Later chapters cut the third leg at the network boundary with an egress proxy, which is why "a credential the agent never holds is a credential it can never leak" becomes the refrain of Part IV.

Why "review the code" isn't the whole answer

Traditional application security leans hard on reading code: audit the source, trace the data flow, prove the dangerous path can't be reached. That work still matters here, but it stops short of the real risk surface, because the dangerous path isn't written down in advance. It's assembled at runtime out of the model's reasoning over content you don't control. You can prove the agent has a Bash tool. You can't prove, from the code, that no combination of inputs will ever lead it to call that tool with a command an attacker supplied.

So the question shifts. Instead of "is this code safe," it becomes "what is this agent able to do, and to whom does that matter, on its worst possible run." You answer that not by trusting the model to always decide correctly, but by constraining what a wrong decision can reach: which files, which network destinations, which credentials, which system capabilities. That reframing is the spine of everything that follows. The next chapter takes inventory of the constraints Claude Code and the Agent SDK already apply before you configure anything, and chapter 3 organizes all of it under three principles you'll use to reason about every later decision.
