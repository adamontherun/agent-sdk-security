Chapter 8 · Part II — Permissions & Settings

Managed Settings & Organization Policy

Everything so far assumed you control the settings file. Across a fleet you don't: developers have their own user settings, projects ship their own .claude, and any of them could try to allow what your policy denies. Managed settings are the layer that can't be argued with. This chapter covers the precedence that decides who wins, the handful of managed-only keys a security operator actually reaches for, why a project's allow rules stay dormant until a human trusts the workspace, and how an embedding host tightens policy through the SDK without ever loosening it.

Precedence: managed sits above the command line

Settings resolve in a fixed precedence, highest authority first: managed settings, then command-line arguments, then local project settings, then shared project settings, then user settings. Managed settings can't be overridden by any other level, including CLI flags. The rule that makes this a security boundary rather than a mere ordering is the one carried over from Chapter 4: if a tool is denied at any level, no other level can allow it. A managed deny can't be undone by --allowedTools, and a user-level deny still blocks a project-level allow, because deny is evaluated before allow across every scope.

PriorityScopeWho writes it
1 (highest)Managed settingsThe organization, via MDM or server-managed settings
2Command-line argumentsThe user, per session
3Local project (.claude/settings.local.json)The user, not committed
4Shared project (.claude/settings.json)The team, committed to the repo
5 (lowest)User (~/.claude/settings.json)The user, across all projects

The lockdown keys worth knowing

Most of the keys below are read only from managed settings: placing them in a user or project file does nothing. The full table is long, so here are the ones a hardening operator reaches for, and what each one buys. One, disableBypassPermissionsMode, is the exception: it works from any scope, but managed is where you pin it so a developer can't set it back.

KeyEffect
allowManagedPermissionRulesOnlyUser and project settings can no longer define allow, ask, or deny rules. Only managed rules apply. Your allowlist becomes the only allowlist.
disableBypassPermissionsModeForbids bypassPermissions mode entirely. (This one works from any scope, but managed is where you pin it so nobody flips it back.)
strictPluginOnlyCustomizationBlocks skills, agents, hooks, and MCP servers from user and project sources, so they can only come from plugins or managed settings. Pass true to lock all four, or an array like ["skills", "hooks"] to lock some.
allowManagedHooksOnlyOnly managed, SDK, and force-enabled-plugin hooks load. User and project hooks are blocked, so a repo can't ship a hook that runs on your developers' machines.
forceRemoteSettingsRefreshBlocks startup until remote managed settings are freshly fetched, and exits if the fetch fails. This is the fail-closed switch: no policy, no session.
sandbox.network.allowManagedDomainsOnlyOnly managed allowedDomains are honored, and non-allowed domains are blocked without prompting. Developers can't widen egress.

Two different guarantees are easy to conflate here, so keep them apart. The first is the ordinary precedence rule, which holds whether or not any managed flag is set: deny beats allow from every scope, so a developer can always tighten policy by adding a deny, and can never widen past a managed deny. That is the one-way ratchet. The second is allowManagedPermissionRulesOnly, which is stricter still. The docs state it "prevents user and project settings from defining allow, ask, or deny permission rules. Only rules in managed settings apply." So under this flag even a developer's own deny is dropped: managed settings become the sole source of every allow, ask, and deny rule. It is a stronger lock than the ratchet, not the same lock.

Watch the precedence resolve

The reference model in solutions/ch08.py resolves a stack of scopes into one effective policy. examples/ch08/precedence_demo.py runs the same adversarial user settings twice, once without the flag and once with it, so the two guarantees are side by side. The developer tries to loosen policy by allowing curl and rm and switching to bypass mode, and also adds a deny of their own.

Case A is the ratchet. The developer's Bash(curl *) allow vanished because a managed deny outranks it, but their Bash(rm *) allow survived, since nothing denies rm and allow rules merge across scopes when the flag is off. Their bypassPermissions default collapsed to default under managed disableBypassPermissionsMode, and their own Read(./.env) deny survived, because tightening is always allowed. A developer can add restrictions and can pick up allows the policy does not forbid; they cannot punch through a managed deny.

Case B adds allowManagedPermissionRulesOnly and the picture changes. The Bash(rm *) allow is gone, and so is the developer's Read(./.env) deny. Only the managed rules remain: managed's Bash(curl *) deny and its Bash(npm run test:*) allow. This is the stronger lock. When you need the fleet's rule set to be exactly what you shipped and nothing a repository or a developer adds, this is the flag that delivers it, at the cost of the developers' ability to tighten locally too.

A hardened managed-settings.json

Putting the keys together, examples/ch08/managed-settings.json is a starting point for a locked-down fleet. It denies the network Bash tools per Chapter 6, keeps git push behind an ask rule, pins the permission-rule and sandbox lockdown flags, refuses to start without fresh policy, and logs settings changes through a ConfigChange hook.

🏭 In production

Lock the policy, then watch it. The security docs recommend auditing or blocking settings changes during a session with ConfigChange hooks, and monitoring fleet activity through OpenTelemetry metrics. A managed policy tells you what's supposed to be true; the ConfigChange hook and OTEL export tell you whether it still is. Chapter 20 builds the observability side of this out.

Project allow rules wait for workspace trust

There is a boundary that protects the individual developer from a malicious repo, and it's easy to miss because it's silent by default. A project's permissions.allow rules and additionalDirectories grant capability, so Claude Code applies them only after you accept the workspace trust dialog for that folder. Until you do, the rules are read but not applied. Clone a hostile repo whose .claude/settings.json allows Bash(*), and none of it takes effect until a human clicks "Yes, I trust this folder." Deny and ask rules aren't gated this way, since they only restrict.

Trust is saved per workspace, keyed on the git repository root. Two edges matter for a deployment. In non-interactive mode with -p, no dialog appears and the untrusted rules simply stay ignored, which is the right default for a headless agent. And starting Claude Code directly in your home directory holds trust for the session only and never writes it to disk, so the prompt returns every launch. Start from a project subdirectory instead.

Embedding hosts tighten through the SDK

If you build a product on top of the Agent SDK, you are the policy authority for your users, and the SDK gives you a managed layer of your own. An embedding host supplies additional managed policy through the managedSettings option, and when parentSettingsBehavior is set to "merge", embedder values can tighten policy but not loosen it. Your host can add deny rules and forbid bypass mode on top of whatever the underlying settings already enforce, and there is no configuration a user can pass that removes what you added. It's the same one-way ratchet as managed settings, exposed to the host you're writing.

🔬 Under the hood

Authority in this chapter flows downward, and that's not a coincidence. Deny merges from all scopes and beats allow, so any scope can tighten and none can widen past a managed deny; workspace trust withholds granted capability until a human consents; the embedder merge tightens but never loosens. allowManagedPermissionRulesOnly is the one control that goes further than the ratchet: it doesn't just stop lower scopes from granting, it stops them from contributing any rule at all, so managed settings become the sole authority rather than the top of a stack. Design your policy assuming every scope below you is adversarial, and the mechanics are on your side.
