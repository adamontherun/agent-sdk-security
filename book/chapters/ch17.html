Chapter 17 · Part V — Production Deployment

Hosting the Agent SDK

Everything up to here hardened one agent: its permissions, its sandbox, the network it can see. Production adds a second question. You are not running one agent, you are running many, often for different tenants, on shared hosts. This chapter covers what a running agent actually is on the box (a supervised subprocess with state on local disk), how long its container should live relative to the sessions it serves, and the isolation settings that keep one tenant's files and memory from bleeding into another's. It is the model that chapters 18 and 19 then place onto two different pieces of AWS.

The subprocess model

When your application calls query(), the SDK does not make an HTTP call and wait. It spawns a separate claude CLI process and talks to it over stdio. That subprocess owns a shell, a working directory, and the JSONL session transcripts on local disk. One agent session is one subprocess, so running N concurrent sessions means N process trees, each holding its own transcript. This is the fact the rest of the chapter follows from, per the hosting documentation.

By default every subprocess inherits your application's working directory. When sessions need separate filesystems, pass cwd on each call so they do not write over each other.

Three kinds of agent state live on the container's filesystem, and none of them survive a container restart, a scale-down, or a move to another node. The documentation names all three:

StateDefault location
Session transcripts~/.claude/projects/, or projects/ under CLAUDE_CONFIG_DIR if set
CLAUDE.md memory files~/.claude/CLAUDE.md for the user tier, the working directory for the project tier
Working-directory artifactsThe session's working directory

A transcript a user expects to resume needs to be mirrored to durable storage with a SessionStore adapter. The store mirrors transcripts only, not memory files or other working-directory artifacts, so those need their own volume or object-store sync. For security this ephemerality is a feature worth keeping where you can: a container that forgets everything on exit is a smaller target than one that accumulates every tenant's history on a shared disk.

🔬 Under the hood

The subprocess is the same claude binary you run interactively, bundled inside the SDK package and pinned to its version. Updating the SDK is how you update the CLI. That also means the telemetry instrumentation from chapter 20 is already compiled into the process the SDK spawns; you turn it on with environment variables, not a separate agent.

Choosing a session pattern

The four patterns in the hosting docs differ on one axis: how long the container lives relative to the sessions it serves. Picking one is mostly a question of how your users show up.

Ephemeral sessions create a container per task and destroy it when the task finishes. A one-shot entrypoint calls the SDK and exits. This fits bug-fix runs, receipt extraction, a document translation, any job with a clear end. It is also the easiest pattern to reason about for security, because the blast radius of a compromised session is one container that is about to be thrown away.

Long-running containers hold many sessions at once behind an HTTP or WebSocket endpoint, mapping each active session to a live query. This is what an email triage agent or a Slack bot needs. In Python you hold a session open across turns with ClaudeSDKClient, and you size the container for the peak number of concurrent sessions it must keep in memory.

Hybrid sessions are ephemeral containers that hydrate from a SessionStore on startup and persist updates back, so they can spin down while idle and come back when the user returns. The store is required here rather than optional: shut a container down without one and the transcript goes with it. This fits work that spans hours but sits idle between check-ins, like a research task a user pokes at now and then.

A multi-agent container runs several SDK subprocesses side by side for agents that collaborate closely. Give each its own working directory and isolate settings loading so per-agent CLAUDE.md files do not leak across agents. That last point is the same isolation problem multi-tenancy raises, which the section below handles directly.

Provisioning and the concurrency math

The docs give 1 GiB of RAM, 5 GiB of disk, and 1 CPU per agent as a starting point for a freshly started instance. Treat it as a floor. Memory grows with session length and tool activity, so a long session under heavy tool load can push well past a gigabyte, and the same host then holds far fewer agents.

A 16 GiB host with 2 GiB of overhead holds fourteen 1 GiB sessions but only four 3 GiB ones, which is the whole point of measuring the ceiling before you size a fleet.

🏭 In production

For long-running containers behind a load balancer, pin each session to one container with consistent hashing on the session ID. A pinned session keeps hitting the same running subprocess until the container recycles it. Two limitations from the docs shape capacity planning: a session has no top-level timeout (bound it with maxTurns), and memory grows over long sessions (cap length or recycle subprocesses periodically).

Multi-tenant isolation

This is the security core of the chapter. By default the SDK reads settings and CLAUDE.md memory files from the filesystem. In a shared container serving more than one tenant, those files are a leak path: one tenant's memory or settings load into another tenant's session. Closing that path takes four SDK-level options plus a network control, all named in the multi-tenant isolation section.

Pass setting_sources=[] so no filesystem settings load at all. Set CLAUDE_CODE_DISABLE_AUTO_MEMORY=1, because auto memory loads into the system prompt regardless of setting_sources. Point CLAUDE_CONFIG_DIR at a per-tenant directory so tenants do not share the global ~/.claude.json. Give each tenant its own working directory and pass cwd explicitly on every call. Construct tenant_dir and config_dir so each tenant gets a path no other tenant can reach. In Python the env dict merges on top of the inherited environment, so the subprocess still sees PATH and ANTHROPIC_API_KEY. The fifth control is the one this course has spent Part IV building: per-tenant egress rules at the proxy, so a compromised tenant cannot exfiltrate through another tenant's outbound policy. Chapters 18 and 19 are where these settings meet real infrastructure.

⚠ Gotcha

setting_sources=[] does not silence every input. Auto memory at ~/.claude/projects/<project>/memory/ loads regardless, which is why CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 is a separate, required step and not a nice-to-have. Set the flag and give each tenant a distinct CLAUDE_CONFIG_DIR; leaving either out reopens the leak you were trying to close.
