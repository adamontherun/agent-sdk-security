Chapter 11 · Part III — Isolation Technologies

gVisor

The hardened container in the last chapter left one boundary standing: it shares the host kernel. When code inside makes a system call, that call goes straight to the same kernel running everything else on the box, so a kernel vulnerability is a path out. gVisor closes that path by putting a second kernel, written in Go and running in userspace, between the container and the real one. This chapter explains how it does that and what it costs, and it is honest about one thing up front: gVisor needs a Linux host, so unlike chapter 10 the commands here were not run on this Mac. The setup is verified against the official docs, and there is a CI workflow you can actually run.

⚠ Can't run this on a Mac

gVisor's runsc runtime requires a Linux host with the kernel features it hooks into. It does not run inside Docker Desktop for Mac's VM without custom modifications this course does not attempt. So the terminal blocks below are the real, exact commands from the gVisor documentation, not output captured on this machine. The runnable form for this chapter is the GitHub Actions workflow at the end, which installs and exercises runsc on an ubuntu-latest runner.

Two processes: the Sentry and the Gofer

gVisor is described in its documentation as an application kernel that implements a Linux-like interface. The part doing the work is the Sentry. The docs call it the largest component and say it can be thought of as an application kernel: it handles system calls, signal delivery, memory management, page faulting, and threading. The important detail is what it does not do. The Sentry does not pass syscalls through to the host kernel; it services them itself. When the application asks for something, the Sentry answers, using a small and tightly restricted set of host calls of its own rather than forwarding the application's request.

Filesystem access is split off into a second process, the Gofer. The docs describe it as a standard host process started with each container that communicates with the Sentry over the 9P protocol on a socket or shared memory channel (9P historically; recent gVisor defaults to its own LISAFS protocol on that channel, though the trust boundary is unchanged). The Sentry itself runs in a restricted seccomp container with no direct filesystem access, so a compromise of the Sentry does not hand an attacker the host's files; it would still have to go through the Gofer's mediated interface. Two processes with narrow, separate jobs is the shape of the design.

🔬 Under the hood

Why write a kernel in Go? Memory safety. The docs point out that unlike Linux, gVisor is written in a memory-safe language and runs in userspace. A large class of kernel exploits depends on memory-corruption bugs in C. A Go implementation of the syscall surface does not remove every risk, but it takes the most common category of kernel vulnerability off the table for the layer the untrusted code talks to first.

How the syscalls get intercepted

For the Sentry to service a syscall, the syscall has to be redirected to it before it reaches the host. gVisor calls the mechanism that does this the Platform, and there have been three. The current default is systrap, which relies on seccomp's SECCOMP_RET_TRAP feature: when the application makes a system call, the kernel sends a SIGSYS signal to the thread, which transfers control to gVisor to handle the call. The docs note systrap replaced ptrace as the default in mid-2023.

The KVM platform is the other supported option. It uses the kernel's KVM virtualization functionality to let the Sentry act as both guest kernel and virtual machine monitor, which can give better performance for address-space switching. The old ptrace platform, which used PTRACE_SYSEMU to run user code without letting it make host syscalls, is deprecated and no longer supported. For deploying an agent, the takeaway is that systrap is what you get by default and it does not require nested virtualization, while KVM is available where the host supports it and the workload benefits.

What it costs

Servicing syscalls in userspace is not free, and the cost depends entirely on what the workload does. The secure-deployment guide gives a compact table, and it matches the qualitative picture in gVisor's own performance guide: CPU-bound work is essentially unaffected because gVisor does not interfere with raw instruction execution, while syscall-heavy and filesystem-heavy work pays a real tax.

The gVisor performance guide fills in why the file-I/O number is a range and not a point. Raw data transfer across the sandbox boundary carries a small fixed overhead, but VFS-heavy patterns, lots of open and close on many small files, hit internal serialization points and degrade sharply. An agent that spends its time thinking, generating text, and making occasional API calls sits near the cheap end. An agent that unpacks a huge dependency tree or walks a giant repository on every run sits near the expensive end. Measure your actual workload before deciding the overhead is or is not acceptable.

🏭 In production

gVisor earns its overhead when you are running code you do not trust next to code, or tenants, you must protect. The secure-deployment guide puts it plainly: for multi-tenant environments or processing untrusted content, the additional isolation is often worth the cost. If a single team runs an agent against its own repositories on its own infrastructure, a hardened container is usually enough and gVisor's file-I/O tax buys little. The decision framework in chapter 13 makes this call explicit against a handful of concrete scenarios.

The verifiable form: a CI workflow

Because this Mac cannot run runsc, the honest "hands-on" for this chapter is a GitHub Actions workflow. An ubuntu-latest runner is a real Linux host, so it can install the runtime and prove a container ran on gVisor's kernel rather than the host's. The file is in the repo; the install steps are the same ones quoted above.

Running uname -a under runsc reports gVisor's own kernel identity, not the host's, which is the confirmation that the container's syscalls were serviced by the Sentry. That is the boundary this chapter is about, made visible in one command.
