Chapter 2 · Part I — Threat Model & Foundations

Built-In Defenses, A Tour

Before you write a single permission rule, Claude Code and the Agent SDK already refuse a lot. They default to read-only, confine writes to the working directory, parse every Bash command before running it, and keep fetched web content out of the main context window. This chapter is a guided tour of those defaults, so that when the later parts start hardening things, you know what you're building on top of rather than reinventing. It also marks the edges of each protection, because a defense you overestimate is worse than one you know the limits of.

Read-only by default

The foundation is the permission model. The Claude Code security documentation states it directly: "Claude Code uses strict read-only permissions by default. When additional actions are needed (editing files, running tests, executing commands), Claude Code requests explicit permission." A built-in set of read-only commands such as ls, cat, and git status runs without a prompt; anything that can modify your system stops and asks.

On top of that sits a write boundary that holds regardless of what the model decides. Claude Code "can only write to the folder where it was started and its subfolders" and "cannot modify files in parent directories without explicit permission." Reads are broader than writes on purpose, so the agent can pull in system libraries and dependencies it needs to understand your code, but a prompt-injected instruction to overwrite ~/.bashrc runs into a wall that isn't a matter of the model's judgment. That asymmetry, wide reads and narrow writes, is the first concrete security boundary in the product, and chapter 16 returns to it when the filesystem gets locked down for real.

The Bash gate: parse first, fail closed

The most security-relevant default is how Bash commands get evaluated. The secure-deployment guide describes it: "Before executing bash commands, Claude Code parses them into an AST and matches the result against your permission rules. Commands that cannot be parsed cleanly, or that do not match an allow rule, require explicit approval." Two behaviors fall out of that, and both are worth internalizing now because the challenge for this chapter models them.

The first is fail-closed matching. The security docs list it plainly: "Unmatched commands default to requiring manual approval." Silence is not consent. If a command doesn't match something you explicitly allowed, it prompts, rather than the reverse where anything not explicitly forbidden slips through. The second is a small set of always-prompt constructs: "A small set of constructs such as eval always require approval regardless of allow rules." Even if you allowlisted it, eval stops, because a command that builds and runs a string at runtime defeats the whole point of matching the command you can see. Related to that, the docs note "command injection detection," where "suspicious bash commands require manual approval even if previously allowlisted."

The demo above allowlists ls, cat, git, and npm. Everything else prompts, and eval prompts even though a careless operator might have added it to the list. Notice curl and wget landing in the prompt column: the security docs call this out as network command approval, where commands that fetch content from the web "are not auto-approved by default." You can still approve them once or add an explicit rule like Bash(curl *), but they don't ride in for free, which matters because a fetch command is a ready-made exfiltration path, the third leg of the trifecta from chapter 1.

⚠ Gotcha

The Bash gate is a permission check, not a safety analysis. The secure-deployment guide is blunt about the distinction: "This is a permission gate, not a sandbox; it does not infer whether a command is dangerous from its target path or effects." An allowed rm rule doesn't get smarter about rm -rf /important versus rm ./scratch.tmp; matching the rule is the whole check. If you need the command's actual effect constrained, that's the sandbox's job (chapter 9) or the isolation boundary's (Part III), not the permission matcher's.

Keeping untrusted content at arm's length

Chapter 1 made the case that untrusted content is a leg of the trifecta. Several defaults exist specifically to keep that content from steering the agent. When the model fetches a web page, it doesn't drop the raw page into the conversation. The security docs describe two pieces here: "Web fetch uses a separate context window to avoid injecting potentially malicious prompts," and search results are handled so that, per the secure-deployment guide, "search results are summarized rather than passing raw content directly into the context, reducing the risk of prompt injection from malicious web content." An instruction buried in a fetched page lands in an isolated context and comes back as a summary, not as a live command sitting next to your own.

Alongside that, the security docs list "context-aware analysis," which "detects potentially harmful instructions by analyzing the full request," and "input sanitization," which "prevents command injection by processing user inputs." And more broadly, "tools that make network requests require user approval by default," so the act of reaching out, not just the act of running a shell fetch, is gated. None of these is a guarantee. They are friction placed deliberately between untrusted text and consequential action, which is exactly the constraint chapter 1 argued for.

Trust verification, and the flag that turns it off

Two entry points into a session get an explicit trust check. The security docs describe "trust verification" covering the case where "first-time codebase runs and new MCP servers require trust verification." The first time you point Claude Code at an unfamiliar directory, or add an MCP server, it asks before proceeding, so a repo you just cloned can't silently run whatever its configuration implies.

🏭 In production

This is the built-in default that most surprises people moving from the interactive CLI to the SDK. The security docs note that "trust verification is disabled when running non-interactively with the -p flag." A headless Agent SDK deployment has no human to answer a trust prompt, so that particular gate is off by design. The protection you were leaning on interactively is simply not there in the automated path, which is the whole reason Parts II through V exist: in production you replace the human-in-the-loop trust check with permission rules, sandboxes, and network controls that don't need anyone at a keyboard.

Source: Claude Code security — Protect against prompt injection

Reducing prompt fatigue without lowering the bar

A defense that asks too often gets clicked through. Claude Code's answer is scoped allowlisting: the security docs describe "prompt fatigue mitigation" as "support for allowlisting frequently used safe commands per-user, per-codebase, or per-organization." Accept Edits mode is the packaged version of this idea, auto-approving file edits and a fixed set of filesystem Bash commands like mkdir, touch, rm, mv, cp, and sed for paths in the working directory, while "other Bash commands and out-of-scope paths still prompt." The point is to make the safe common case quiet so the genuinely unusual request still stands out. Part II takes this apart in detail, since the exact boundary of what each mode auto-approves is where a lot of real deployments get their risk model wrong.

⏭ Version watch

The sandboxed Bash tool, enabled with /sandbox, is the one default on this tour that is still visibly evolving, and its limits are documented as active. The secure-deployment guide notes the built-in sandbox does no TLS inspection and can be bypassed by techniques like domain fronting, pointing readers to a TLS-terminating proxy where the threat model demands it. Treat sandbox mode as a strong default that raises the bar, not a settled guarantee, and check the sandboxing documentation for its current limitations before relying on it. Chapter 9 covers this boundary in full.

That's the inventory. Read-only defaults, a write boundary that ignores the model's opinion, a Bash gate that parses and fails closed, web content held in a separate context, trust checks at first contact, and allowlisting to keep the prompts meaningful. Every one of them has an edge, and naming those edges is the point of the tour: the rest of the course is about the controls you add where these stop. Chapter 3 gives you the three principles to organize all of it.
