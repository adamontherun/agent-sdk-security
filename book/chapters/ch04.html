Chapter 4 · Part II — Permissions & Settings

The Permission Evaluation Pipeline

Every permission rule you write is a bet about the order things are checked in. Write a careful allow rule and a careless deny rule in the same file, and the deny wins even when it's broader and less specific. This chapter traces one tool call from the moment Claude requests it to the moment it runs or is refused, in both Claude Code and the Agent SDK, so that when you harden a deployment later you know exactly which layer each rule lands in.

Claude Code checks deny, then ask, then allow

Claude Code sorts its permission rules into three buckets and consults them in a fixed order. Deny rules are checked first, then ask rules, then allow rules, and the first bucket that matches decides the outcome. The documentation states the consequence plainly: rules are evaluated in order, deny then ask then allow, and rule specificity doesn't change the order. A one-line exact-match allow does not outrank a sloppy wildcard deny. Specificity is a property people expect to matter because CSS and routing tables work that way. Here it doesn't.

That single design choice is what makes deny rules a dependable security control. If you write Bash(aws *) in the deny list, no allow rule anywhere can carve out an exception for Bash(aws s3 ls), because the deny bucket is exhausted before the allow bucket is ever read. The same holds between ask and allow: a matching ask rule still prompts even when a more specific allow rule would have approved the call silently. A deny is a wall, not a default you can override from a higher-priority file.

🏭 In production

This is the property to lean on when you can't enumerate everything an agent might try. You will never write an allowlist that anticipates every dangerous command. You can write a short deny list for the handful of capabilities that must never be reachable, and trust that nothing downstream can loosen it. Chapter 8 shows how to pin that deny list in managed settings so a developer's local file can't remove it.

A bare tool name behaves differently from a scoped pattern

Denying a tool has two distinct effects depending on how you write it. A bare tool name in the deny list, like Bash, removes the tool from Claude's context entirely. Claude never sees it in the tool list and can't attempt it. A scoped pattern like Bash(rm *) leaves the tool available and blocks only the calls that match when Claude tries them. The docs are explicit that a bare tool name removes the tool from Claude's context, while a scoped rule leaves the tool available and blocks matching calls, and that Bash(*) is treated the same as the bare name.

The distinction matters for how the model behaves, not just whether a call succeeds. Remove Bash entirely and Claude plans around its absence from the start. Leave Bash present but block rm *, and Claude may still propose an rm, watch it get refused, and try to route around the refusal. Neither is wrong, but they lead to different agent behaviour, and a security review should know which one a given rule produces.

The Agent SDK runs six steps in a fixed order

The SDK exposes the same deny-ask-allow precedence, wrapped in a larger pipeline. When Claude requests a tool, the SDK checks permissions in six ordered steps:

StepWhat it does
1. HooksPreToolUse hooks run first. A hook can deny outright or pass the call on. A hook that returns allow does not skip the deny and ask rules below.
2. Deny rulesFrom disallowed_tools and settings. A match blocks the call in every mode, including bypassPermissions.
3. Ask rulesFrom settings. A match falls through to your canUseTool callback for confirmation, even under bypassPermissions.
4. Permission modebypassPermissions approves what reaches here; acceptEdits approves file operations; plan routes writes to the callback. Other modes fall through.
5. Allow rulesFrom allowed_tools and settings. A match approves the call.
6. canUseToolIf nothing above resolved the call, the runtime callback decides. In dontAsk mode this step is skipped and the call is denied.

Reading the steps top to bottom answers most of the questions people ask about the SDK. Why does a deny rule fire even in bypass mode? Because step 2 runs before step 4. Why does an ask rule still prompt when I've allowed the tool? Because step 3 runs before step 5. The order is the whole specification.

When your canUseTool callback can never run

The callback at step 6 is where teams often put their real authorization logic: a check against an external policy service, a per-user allowlist, an audit write. The trap is that earlier steps can approve a call before step 6 is ever reached, and then the callback silently never runs for that tool. The SDK calls this shadowing. As of v2.1.198 the TypeScript SDK emits a one-time Node warning whose code is CLAUDE_SDK_CAN_USE_TOOL_SHADOWED when it detects a configuration the callback can't be reached in.

Two configurations trigger it. First, permissionMode: 'bypassPermissions', which approves every call at step 4. Second, any bare allowedTools entry such as "Read", which auto-approves that whole tool at step 5. A scoped entry like Bash(ls *) does not shadow, because non-matching Bash calls still fall through to the callback. The tail of the demo above checks exactly these cases:

⚠ Gotcha

If your security check lives in canUseTool and you also set a bare allow like allowed_tools=["Read"], reads sail through unchecked. The fix the docs recommend is to move any check that must run on every call into a PreToolUse hook instead, since hooks run at step 1 before anything can approve the call. Treat the CLAUDE_SDK_CAN_USE_TOOL_SHADOWED warning as a real finding, not console noise.

Source: Agent SDK permissions — canUseTool shadowing

Where the sandbox changes step 4

One interaction is worth flagging now because it changes what step 4 does. When the Bash sandbox is enabled with autoAllowBashIfSandboxed: true, which is the default, sandboxed Bash commands run without prompting even if you have a bare Bash ask rule, because the sandbox boundary substitutes for that whole-tool prompt. The exceptions still hold: content-scoped ask rules like Bash(git push *) still prompt, explicit deny rules still apply, and removals targeting / or your home directory still prompt as a circuit breaker. The sandbox is Part III's subject; the point here is only that enabling it quietly weakens a bare-Bash ask rule, so don't rely on that rule alone as your prompt gate.

🔬 Under the hood

Permission rules are enforced by Claude Code, not by the model. Instructions in your prompt or CLAUDE.md shape what Claude tries to do, but they never change what the pipeline allows. That separation is the reason a prompt injection can change the agent's intent without changing its capabilities: the six steps run identically whether the model's plan came from you or from a malicious web page it just read.
