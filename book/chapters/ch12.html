Chapter 12 · Part III — Isolation Technologies

MicroVMs and Firecracker

gVisor gave the agent its own kernel in software. A virtual machine gives it its own kernel in hardware, behind the CPU's virtualization extensions, which is the strongest boundary in this part. The old objection to VMs was weight: seconds to boot and hundreds of megabytes of overhead made them impractical to spin up per task. Firecracker is the answer to that objection. It boots a stripped-down virtual machine in a fraction of a second with a few megabytes of overhead, which makes VM-grade isolation cheap enough to use per agent session. As with gVisor, this needs Linux and KVM, so the configs here are verified against the official docs rather than run on this Mac.

⚠ Can't run this on a Mac

Firecracker is a Linux VMM built on KVM. It does not run on this macOS build machine, which has no KVM. The JSON and commands below are real, taken from the Firecracker project's own documentation and API schema, and the field names are checked against the OpenAPI spec in the repository. What is verified here is the config and the citations; actually booting a microVM needs a Linux host with /dev/kvm.

A deliberately small machine

The Firecracker project describes itself as a virtual machine monitor, a VMM, that uses the Linux Kernel-based Virtual Machine (KVM) to create and manage microVMs. Firecracker is written in Rust, which gives the VMM itself the same memory-safety argument gVisor's Go kernel has. The design decision that makes it a microVM rather than a general-purpose VM is what it leaves out. Where a traditional hypervisor emulates a whole PC's worth of devices, Firecracker emulates only five: virtio-net, virtio-block, virtio-vsock, a serial console, and a minimal keyboard controller used only to stop the microVM. Every device that is not there is attack surface that is not there.

That minimalism is what buys the numbers. The project states Firecracker initiates user-space or application code in as little as 125 ms, runs each microVM with a memory overhead of less than 5 MiB, and supports creation rates of up to 150 microVMs per second per host. Those figures are why per-session VM isolation is realistic: booting a fresh VM for each agent task is on the order of a tenth of a second, not ten seconds.

🔬 Under the hood

Firecracker pairs the hardware boundary with a userspace one. The getting-started guide says the VMM is designed to run inside an execution jail set up by the jailer binary. The project's design documentation presents that jail as an additional layer of defense in depth: it applies the ordinary Linux userspace security barriers, namespaces, cgroups, a chroot, dropped privileges, around the Firecracker process, so an escape from the guest still lands inside a locked-down host process rather than on the host proper.

vsock instead of a network card

The isolation pattern for an agent looks like the --network none container from chapter 10, taken one step further. The agent's microVM is configured with no external network interface at all. Instead it gets a virtio-vsock device, a virtual socket that connects the guest to the host and nothing else. All traffic routes over that vsock to a proxy on the host, which enforces allowlists and injects credentials before forwarding anything outward. There is no NIC for a compromised agent to send packets from, so the proxy is not one option for egress; it is the only path.

Configuring a microVM is a sequence of API calls to a local socket, or a single config file passed at launch. The vsock body has two fields, verified against the project's vsock documentation: a guest_cid and the uds_path of the host-side Unix socket the proxy listens on.

The empty network-interfaces array is the whole security posture in one line. No network-interfaces entry means no virtio-net device is created, which means the guest has no way to reach a network directly. The equivalent API-call form, one PUT per resource, is in examples/ch12/configure-microvm.sh; the boot-source, drive, and action bodies come straight from the getting-started guide.

The primitive underneath the managed services

Firecracker was built at AWS to improve the experience of Lambda and Fargate, and that lineage is the practical reason it matters here. Most teams will never run Firecracker directly. They will consume it through a service that runs it for them. AWS built Lambda and Fargate on this mechanism, and it now offers the same isolation as a directly consumable primitive: AWS Lambda MicroVMs, which launches a Firecracker microVM per session and gives you lifecycle control over it. When you use that service, the vsock-to-proxy pattern, the per-session boot, and the minimal device model described above are what is running for you underneath.

🏭 In production

The build-versus-consume decision is usually consume. Running Firecracker yourself means managing kernel and rootfs images, the jailer, the vsock proxy, and the orchestration that pools and recycles microVMs. That is real engineering, justified when you are a platform provider running untrusted tenant code at scale. For everyone else, the same isolation arrives managed: chapter 19 deploys an agent on AWS Lambda MicroVMs and works through the image-then-launch model, per-session isolation, and when it beats a long-running container. This chapter is the mechanism; that chapter is the product built on it.

Where this leaves the strength ladder

Across four chapters the boundary has climbed a clear ladder. The built-in sandbox confines Bash on a shared kernel. A hardened container adds namespaces, dropped capabilities, and resource limits, still on a shared kernel. gVisor replaces the shared kernel with a userspace one for the syscalls the agent makes. A microVM gives the agent a real, separate kernel behind hardware virtualization, with a five-device attack surface and a jailer around the VMM. Each step is stronger and costs more, in overhead, in complexity, or in both. Knowing the ladder is not the same as knowing which rung your deployment belongs on, which is exactly what the next chapter decides.

⚡ Try it

No Linux host with KVM? The configs in examples/ch12/ are reference material you can read and adapt; booting them needs /dev/kvm, which this course does not require you to have.

The challenge in challenges/ch12/ is runnable anywhere. You implement validate_microvm_config(cfg), which checks a Firecracker config for the no-external-NIC isolation pattern: a real kernel and root drive, sized CPU and memory, a vsock, and above all no populated network-interfaces. Verify with COURSE_SOLUTIONS=1 uv run pytest challenges/ch12. One test adds a network interface and expects external-nic-present back, because a NIC breaks the pattern this chapter is built on.
