Chapter 18 · Part V — Production Deployment

Deploying on Amazon ECS

Chapter 17 said the agent runs in a container. This chapter puts that container on Amazon ECS Fargate and wires it up the way this course argues it should be: no direct route to the internet, a Squid egress proxy running beside it as a sidecar, and two IAM roles that each hold only what they need. The whole thing is real Terraform in examples/ch18/terraform/, and it passes terraform validate and terraform fmt -check against Terraform 1.10.4 with the actual AWS provider. You will not deploy it here (that needs an AWS account), but every resource is schema-valid and ready to plan against one.

The architecture, and why this shape

The Agent SDK secure-deployment guide gives a five-point recipe for cloud deployments: run agent containers in a private subnet with no internet gateway, block all egress except to your proxy with cloud firewall rules, run a proxy that enforces domain allowlists and injects credentials, assign minimal IAM to the agent's service account, and log all traffic at the proxy. The Terraform here implements that recipe on ECS, and the proxy is the Squid you built in chapter 15. Those five points are quoted directly from the guide's Cloud deployments section.

On Fargate the proxy runs as a sidecar container in the same task as the agent. That works because of how Fargate does networking. Each task gets one elastic network interface, and the containers in a task share a network namespace, so they reach each other over loopback. The ECS documentation is explicit: "Inside a task, containers that belong to the task can inter-communicate over localhost." The agent container sets HTTP_PROXY=http://localhost:3128, and the Squid sidecar answers on that port with no service discovery involved.

🔬 Under the hood

The same shared namespace that makes localhost work also means Fargate cannot network-isolate one container in a task from another; they share the task ENI and its security group. So the sidecar is a cooperative egress point: the agent's HTTP client honors HTTP_PROXY and sends everything to Squid, and the task security group is a coarse backstop that limits which ports can leave the ENI at all. If your threat model needs the proxy to be unbypassable, move it to its own task in its own subnet and let the security group allow the agent egress only to the proxy's address. The sidecar buys simplicity; the separate task buys enforcement. This chapter ships the sidecar and flags the trade rather than pretending the sidecar is airtight.

Source: Amazon ECS task networking options for Fargate

The task definition

Two containers, one task. The agent holds no credentials and points its HTTP client at the sidecar; the sidecar holds the injected key and forwards only allowlisted domains. The agent also waits for the proxy to report healthy before it starts, so its first outbound call does not race the sidecar.

The secrets field is the important detail. ECS reads the Secrets Manager ARN at task start and injects the value as an environment variable inside the sidecar. The plaintext key never lands in the task definition JSON, so it does not show up in describe-task-definition output, in the console, or in your Terraform state as a literal. That is the difference between the secrets field and putting the key in environment, which the chapter 18 challenge exists to catch.

Two IAM roles, two jobs

ECS tasks carry two roles, and conflating them is a common way to hand the agent more than it needs. The execution role belongs to the ECS agent that starts the task: it pulls the image, writes logs, and resolves the secret ARN. The task role belongs to the code running inside the container. Keeping them separate means the permissions the platform needs to launch the task are never granted to the process the agent runs.

A task role with no attached policies looks wrong the first time you write it, and it is exactly right here. The agent gets its Anthropic access through the sidecar, not through an AWS credential, so there is nothing the task role needs to grant. If a later feature needs the agent to read one S3 object, that one action gets added to this role and nothing else does.

🏭 In production

Fargate already removes several sharp edges for you. Each task runs on isolated infrastructure and does not share network interfaces, storage, CPU, or memory with other tasks; privileged containers are not available; and capabilities like CAP_SYS_ADMIN and CAP_NET_ADMIN are restricted to prevent privilege escalation. Neither you nor an AWS operator can connect to the host running the workload. You still owe the network and IAM controls above, but the container-escape surface is smaller than on self-managed EC2.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Terraform and the AWS CLI are already installed, and validation needs no AWS credentials.

From examples/ch18/terraform/, run terraform init -backend=false, then terraform fmt -check and terraform validate. Read network.tf to see the egress-restricted security group, iam.tf for the two roles, and ecs.tf for the sidecar wiring. The Squid image is built from examples/ch18/squid/.

Then implement audit_task_definition in challenges/ch18/challenge.py: it flags a task definition that shares one role for both jobs, runs a privileged container, or puts a secret in plaintext environment instead of the secrets field. Check it with uv run pytest challenges/ch18, then COURSE_SOLUTIONS=1 uv run pytest challenges/ch18.
