Chapter 10 · Part III — Isolation Technologies

Hardened Containers

A container out of the box shares a lot with its host: a full set of Linux capabilities, a writable root filesystem, a network interface, no limit on processes or memory. A hardened container gives back almost all of that. In this chapter you build one by hand from the exact flag set the secure-deployment guide recommends, run it on this machine, and prove with captured terminal output that each flag actually stops something. The proofs are the point. It is easy to copy a long docker run line; it is worth seeing the kernel refuse a write, a connection, and a privileged operation because of it.

The flag set

Here is the hardened configuration from the secure-deployment documentation, verbatim. Every line does one job, and the sections below prove four of them on real hardware.

Each option maps to a specific reduction in what a process inside can do. --cap-drop ALL removes Linux capabilities such as NET_ADMIN and SYS_ADMIN that could enable privilege escalation. --security-opt no-new-privileges prevents a process from gaining privileges through a setuid binary. --security-opt seccomp= restricts the available syscalls; Docker's default profile already blocks around 44, and a custom profile can block more. --read-only makes the container's root filesystem immutable, so the agent cannot persist changes, while the --tmpfs mounts give it writable scratch space that is wiped when the container stops. --network none removes every network interface, leaving the mounted Unix socket as the only way out. --memory, --cpus, and --pids-limit cap resources so a runaway or a fork bomb cannot exhaust the host. --user 1000:1000 runs the process as a non-root user, and the read-only code mount lets the agent analyze code it cannot change.

⚠ Gotcha

The read-only code mount is where credentials leak into a hardened container. Even :ro access exposes anything sitting in the directory you mount. The docs list the files to exclude or sanitize first: .env and .env.local, ~/.git-credentials, ~/.aws/credentials, ~/.config/gcloud/application_default_credentials.json, ~/.azure/, ~/.docker/config.json, ~/.kube/config, .npmrc and .pypirc, any *-service-account.json, and *.pem or *.key private keys. Mount only the source you need, or filter with a .dockerignore-style copy step.

Source: Secure deployment · read-only code mounting

A payload to confine

To prove anything, you need something running inside that reports what it can and cannot do. The image is deliberately minimal: a Python base, a non-root user, and one script. That script is not the Agent SDK. It is a set of probes standing in for whatever an agent, or a prompt-injected command an agent ran, might attempt from inside the box.

Build it once with a clearly namespaced tag so it will not collide with anything else on the daemon:

Proof 1: --read-only blocks writes outside a tmpfs

Run the image with a read-only root and a single writable /tmp tmpfs, then try to write to the application directory on the root filesystem. The kernel refuses. Writing to the tmpfs mount succeeds. This is the same command twice with a different target path.

Proof 2: --network none removes every interface

List the network interfaces with and without the flag. The default run has an eth0; the isolated run does not, leaving only loopback and the kernel's always-present virtual tunnel devices. An outbound connection under --network none then fails because there is no route to anywhere.

The eth0 present in the first list is gone from the second. With no interface, the only way for the agent to reach the outside world is the Unix socket you mount, which connects to a proxy on the host. That proxy is where domain allowlisting, credential injection, and traffic logging live, and it is the whole subject of Part IV. A compromised agent behind --network none cannot exfiltrate to an arbitrary server; it can only speak to whatever is on the far end of that socket.

Proof 3: --cap-drop ALL removes CAP_CHOWN

Capabilities are easiest to see with one that root normally has and a hardened container does not. Changing a file's owner needs CAP_CHOWN. Run as root with default capabilities and the chown succeeds. Run as root with --cap-drop ALL and the same operation is refused, even though the process is uid 0.

Being root inside the container is not the same as being able to do root-level things. Dropping capabilities is what makes a non-root user redundant as a defense rather than the only line of it.

Proof 4: --pids-limit caps the process table

A fork bomb is the classic denial-of-service a shared host worries about. With --pids-limit 16, once the container is holding sixteen processes the kernel refuses to create more, so a loop asking for forty background sleeps hits a wall. The same workload without the limit spawns them all without complaint.

The whole thing, self-reporting

Put every flag together and let the payload report from inside. This is the hardened container running end to end: non-root, read-only root with a tmpfs, no network, capped memory, CPU, and process count, and private IPC. The output is the agent describing its own confinement.

🏭 In production

Two more flags harden the boundary further. --userns-remap maps the container's root to an unprivileged user on the host, so even a container escape lands as a nobody rather than host root; it needs daemon configuration to enable. --ipc private gives the container its own IPC namespace so it cannot reach another container's shared memory. Neither shows up in a quick smoke test the way the four proofs above do, but both matter when containers share a host with workloads they should not be able to observe.

Source: Docker · isolate containers with a user namespace

🔬 Under the hood

These proofs ran on macOS with Docker Desktop, which is worth being precise about. Docker Desktop runs a real Linux VM, and the container runs inside that VM's Linux kernel. So --cap-drop, --read-only, --network none, and --pids-limit are enforced by a genuine Linux kernel, not emulated. What that same kernel also means is the limitation the next chapter opens on: your container and every other container share that one kernel. A hardened container shrinks the attack surface a great deal, but a kernel vulnerability is still a path across the boundary.
