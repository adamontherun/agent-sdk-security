Chapter 6 · Part II — Permissions & Settings

Tool-Specific Rules: Bash, Files, Web, MCP

A permission rule is a small pattern-matching language, and every tool speaks a slightly different dialect of it. Bash rules match command strings with word-boundary wildcards; Read and Edit rules follow gitignore anchoring; WebFetch matches hostnames with a rule about crossing dots. Get a dialect wrong and you write a rule that looks like a boundary but leaks. This is the densest chapter in the part, so it leans on tables and worked matches, and it ends with a challenge that makes you implement the Bash matcher yourself.

Bash: the space before the star

Bash rules match with * wildcards that can appear anywhere in the command. The subtlety that trips everyone is the word boundary. As the docs put it, the space before * matters: Bash(ls *) matches ls -la but not lsof, while Bash(ls*) matches both. A trailing " *" requires the prefix to be followed by a space or the end of the string, so it can't spill into a longer command name. Drop the space and you've written a prefix match that also grants lsof, lscpu, and anything else starting with ls. The :* suffix is an equivalent way to write the trailing wildcard, so Bash(ls:*) and Bash(ls *) mean the same thing.

RuleMatchesDoes not match
Bash(ls *)ls -la, lslsof
Bash(ls*)ls -la, lsof—
Bash(git * main)git checkout main, git push origin maingit checkout dev
Bash(* --version)python --versionpython --help
Bash(npm run build)npm run buildnpm run build --prod

Compound commands and process wrappers

Claude Code is aware of shell operators, so a rule doesn't leak across a &&. The recognized separators are &&, ||, ;, |, |&, &, and newlines, and a rule must match each subcommand independently. This is the difference between a rule and a bypass. Bash(git status) does not authorize git status && rm -rf /, because the rm -rf / half was never matched by anything.

Before matching, Claude Code strips a fixed set of process wrappers, so Bash(npm test *) also matches timeout 30 npm test. The stripped wrappers are timeout, time, nice, nohup, and stdbuf, plus bare xargs with no flags. The list is built in and not configurable, and it deliberately excludes environment runners like npx, docker exec, and devbox run. Those execute their arguments as a command, so a rule like Bash(devbox run *) would match devbox run rm -rf .. If you allow work inside a runner, name the inner command too: Bash(devbox run npm test).

The curl-by-URL rule that looks safe but isn't

Here is the rule people write and shouldn't. You want to let the agent fetch from GitHub, so you write Bash(curl http://github.com/ *) and feel covered. You aren't. The docs list the variations that slip past it, and they're not exotic:

VariationWhy the rule misses it
curl -X GET http://github.com/...An option sits before the URL, so the prefix doesn't match.
curl https://github.com/...Different protocol.
curl -L http://bit.ly/xyzA redirect that lands on GitHub, or anywhere else.
URL=http://github.com && curl $URLThe URL is in a variable, not the literal command.
curl  http://github.comExtra whitespace breaks the exact prefix.

The docs are blunt that Bash permission patterns that try to constrain command arguments are fragile, and they recommend a different shape entirely. Block the network tools with deny rules, Bash(curl:*) and Bash(wget:*), then grant the specific domains through the WebFetch tool with WebFetch(domain:github.com). Or validate URLs in a PreToolUse hook, which sees the whole command string and can parse every URL out of it. Chapter 7 builds exactly that hook. The rule of thumb: a permission rule can gate a command reliably, but it can't gate the arguments to a command that reaches the network.

⚠ Gotcha

Allowing WebFetch to a domain does not stop Bash from reaching it. The docs note that if Bash is allowed, Claude can still use curl or wget to hit any URL. The two tools are gated separately. To actually confine egress you need both the Bash deny rules and, ultimately, a network boundary the agent can't route around, which is Part IV.

Read and Edit: four ways to anchor a path

File rules follow the gitignore specification, with four anchor styles that resolve to very different places. The one that catches people is the single leading slash: it is not an absolute path.

PatternAnchorExample resolves to
//pathFilesystem rootRead(//Users/alice/secrets/**) → /Users/alice/secrets/**
~/pathHome directoryRead(~/Documents/*.pdf) → home /Documents/*.pdf
/pathThe settings file's own locationEdit(/src/**) in project settings → <project root>/src/**
path or ./pathCurrent directoryRead(*.env) → <cwd>/*.env

A single-slash /path anchors at whatever settings file defines it, which is why a deny rule like Read(/secrets/**) written in user settings blocks ~/.claude/secrets/**, not a secrets directory in your project. To write a user-settings rule that applies inside every project, use a // absolute path or a ~/ home-relative one. Bare filenames match at any depth, so Read(.env) and Read(**/.env) are equivalent and block every .env at or under the current directory.

Read and Edit deny rules reach further than the built-in tools. They also apply to file commands Claude Code recognizes in Bash, like cat, head, tail, and sed. They do not apply to arbitrary subprocesses, so a Python or Node script that opens a file itself slips past them. For a boundary that holds against any process, the docs point at the sandbox, which is Part III.

🔬 Under the hood

Symlinks are checked twice, and allow and deny rules treat the pair asymmetrically. An allow rule applies only when both the symlink path and its target match, so a symlink inside an allowed directory that points outside it still prompts. A deny rule applies when either the link or its target matches, so a symlink pointing at a denied file is itself denied. With Read(./project/**) allowed and Read(~/.ssh/**) denied, a link at ./project/key pointing to ~/.ssh/id_rsa is blocked: the target fails the allow and matches the deny. The asymmetry is deliberate, and it favours safety.

WebFetch: a wildcard can't cross a dot

WebFetch rules match against the hostname of the requested URL with a domain: prefix. The security-relevant rule is how far a wildcard reaches. A leading *. matches subdomains at any depth, so WebFetch(domain:*.example.com) matches api.example.com and a.b.example.com but not example.com itself. In any other position, the wildcard matches only the text between two dots. So WebFetch(domain:example.*) matches example.org, where * becomes org, but not example.evil.com, where * would have to swallow evil.com and cross a dot to do it. That constraint is what stops a trailing wildcard from matching an attacker-registered lookalike.

MCP and subagent rules

MCP rules name a server, optionally with a tool. mcp__puppeteer matches any tool from the puppeteer server; mcp__puppeteer__* does the same with an explicit wildcard; and mcp__puppeteer__puppeteer_navigate matches one tool. Allow rules accept a tool-name wildcard only after a literal mcp__<server>__ prefix, so the server segment must be glob-free. An unanchored allow glob like "*" or "mcp__*" is skipped with a warning and grants nothing, though as a deny rule mcp__* does remove every MCP tool.

Subagents get their own rule family. Agent(Explore) matches the Explore subagent, Agent(Plan) the Plan subagent, and Agent(my-custom-agent) a custom one. Put these in the deny list to disable a subagent an agent shouldn't be able to spawn. Recent builds also allow matching a top-level input parameter on any tool, so Agent(model:opus) matches Agent calls that request the Opus tier and Bash(run_in_background:true) matches backgrounded Bash calls, which gives you another axis to gate on.
