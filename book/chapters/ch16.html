Chapter 16 · Part IV — Credentials & the Network Boundary

Filesystem Configuration

The network boundary from the last two chapters keeps a credential from leaving. The filesystem boundary keeps one from being present in the first place. An agent that mounts your source tree read-only still reads every file in it, and source trees are full of secrets that were never meant to travel: a stray .env, an ~/.aws/credentials a script left behind, a private key in a certs/ directory. This chapter is the reference for what to mount, what to mount read-only, and what to keep out entirely, plus a pre-flight scanner you can run before any directory becomes a mount.

Read-only is the floor, not the ceiling

When the agent analyzes code but should not change it, mount the directory read-only. In Docker that is the :ro suffix on the volume, and it is the same mount chapter 10's hardened container used for /workspace.

Read-only stops the agent from modifying the code. It does nothing about reading. As the secure-deployment guide warns, even read-only access to a code directory can expose credentials, and it gives a specific list of files to exclude or sanitize before mounting. The risk in each case is not subtle: these are files that hold a working credential in plaintext.

The exclusion table

This is the guide's list, reproduced so you can check a tree against it. If a mount would include any of these, either the mount is scoped too broadly or the file needs to be removed from what you copy in.

FileWhat it leaks
.env, .env.localAPI keys, database passwords, secrets
~/.git-credentialsGit passwords and tokens in plaintext
~/.aws/credentialsAWS access keys
~/.config/gcloud/application_default_credentials.jsonGoogle Cloud ADC tokens
~/.azure/Azure CLI credentials
~/.docker/config.jsonDocker registry auth tokens
~/.kube/configKubernetes cluster credentials
.npmrc, .pypircPackage registry tokens
*-service-account.jsonGCP service account keys
*.pem, *.keyPrivate keys

The guide's advice is to copy only the source files you actually need, or apply .dockerignore-style filtering, rather than mounting a home directory or a repo root and hoping it is clean. A scanner that walks the tree and flags these patterns before you wire up the mount turns that advice into something you can put in a build step, which is the example for this chapter.

⚠ Gotcha

The dangerous files are often the ones a glob written for source code skips. A pattern like *.py or src/** looks safe precisely because it never mentions .env or *.key, so the review that approves it never sees them either. The scanner below inverts that: it looks only for the files you do not want, so a clean run is positive evidence rather than the absence of a warning. It reads filenames and paths only, never file contents, so it is safe to point at a tree full of live secrets.

Where the agent is allowed to write

If the agent needs to write, the question is whether those changes should survive the container, and the answer picks the mechanism. The secure-deployment guide lays out three cases.

For scratch space that should vanish when the container stops, use a tmpfs mount. It lives in memory and is cleared on stop, and it pairs with a --read-only root so the only writable places are the ones you named. This is the ephemeral-workspace pattern from chapter 10.

When you want to review changes before they are real, an overlay filesystem lets the agent write into a separate upper layer without touching the files underneath. You inspect that layer afterward and apply or discard it, which is the natural fit for an agent whose output you do not yet trust. For output that genuinely needs to persist, mount a dedicated volume, and keep it separate from anything sensitive so a write the agent makes cannot land next to a credential.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — the repo and Python environment are already set up.

Run the pre-flight scanner against any directory: python3 examples/ch16/preflight_scan.py /path/to/code. It walks the tree, prints every file matching the exclusion table, and exits non-zero if it finds any, so it can gate a mount in a build script or CI job. Point it at a temp directory with a planted .env and watch it refuse.

Then open challenges/ch16/ and implement scan_for_secrets until the tests pass: uv run pytest challenges/ch16. The test builds a real tmp_path tree mixing credential files with innocuous ones (including a kube-config-guide.txt that must not be flagged just because its name contains the word), and checks you return exactly the sensitive paths, sorted.
