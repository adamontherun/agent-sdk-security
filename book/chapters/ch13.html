Chapter 13 · Part III — Isolation Technologies

Choosing an Isolation Strategy

Four chapters, four boundaries, each stronger and more expensive than the last. The mistake this chapter exists to prevent is picking by strength alone and paying for isolation your threat model does not need, or worse, picking by cost alone and shipping a boundary that does not hold against the content you actually process. The right choice falls out of two questions: whose code runs inside the boundary, and what happens if it turns hostile. This chapter turns those questions into a decision you can defend.

The tradeoff, in one table

The secure-deployment guide summarizes the four options along the three axes that matter for the choice. Read it as the shape of the tradeoff, not the answer; the answer depends on the scenario, which the rest of the chapter works through.

TechnologyIsolation strengthPerformance overheadComplexity
Sandbox runtimeGood (secure defaults)Very lowLow
Containers (Docker)Setup dependentLowMedium
gVisorExcellent (with correct setup)Medium/HighMedium
VMs (Firecracker, QEMU)Excellent (with correct setup)HighMedium/High

Source: Secure deployment · isolation technologies

Two entries in that table carry a parenthetical that does the real work. Containers are "setup dependent," which chapter 10 made concrete: an unhardened docker run shares almost everything with the host, and it was the specific flags, the dropped capabilities, the read-only root, the removed network, that turned it into a boundary. gVisor and VMs are excellent "with correct setup," which chapters 11 and 12 showed means a Linux host, a registered runtime or a KVM device, and the config to match. Strength on paper is not strength deployed.

The decision, as a tree

Work top to bottom and stop at the first branch that fits. Each question is about your threat model, not your preferences, and the order encodes which risks justify which cost.

The ordering is deliberate. Kernel isolation as a hard requirement sits at the top because when it is genuinely required, nothing cheaper qualifies, no matter how convenient. Multi-tenancy plus regulated data comes next because the blast radius of a failure is another customer's protected information. Untrusted content without the regulatory weight drops to gVisor, which buys the separate kernel that chapter 11 described without the operational load of running VMs. Everything that is your own code, run for your own benefit, lives in the bottom two rungs, where the question is only how automated the run is.

Four scenarios, worked

A single developer on a laptop

One person runs Claude Code against their own repositories. The code is theirs, the machine is theirs, and the realistic risk is a prompt-injected command in a README doing something the developer did not intend, not a determined attacker with a kernel exploit. This is the built-in sandbox's home ground. One settings block, the OS-level filesystem and network confinement from chapter 9, and the credential deny list to close the read-by-default gap. Reaching for a container here adds friction for a threat the sandbox already covers. The one thing not to skip is the sandbox.credentials block, because the default that still exposes ~/.aws/credentials is the realistic failure here, not a kernel escape.

A CI pipeline running agent tasks

Agent jobs run unattended on shared CI infrastructure, checking out branches and running builds. The code is still trusted, mostly, but nobody is watching each run, jobs share a host with other jobs, and a runaway task should not take the runner down. This is the hardened container's case. The resource caps from chapter 10, --memory, --cpus, and the --pids-limit that refused a fork bomb, keep one job from starving another. --read-only with a tmpfs keeps a job from persisting changes into the runner, and --network none with a proxy socket keeps an unattended job from reaching hosts it has no business reaching. The container is not overkill here; the lack of a human in the loop is exactly what the resource and network limits stand in for.

Multi-tenant SaaS processing customer-uploaded content

Your product runs an agent over content your customers upload, and one customer's document might contain instructions aimed at attacking another customer or your platform. Now the code path is deliberately exposed to hostile input, and tenants share infrastructure. A shared kernel is the risk that matters, because a container escape through a kernel bug crosses tenant boundaries. gVisor is the fit: its userspace kernel means the untrusted code would have to defeat the Sentry's Go implementation before it got near the host kernel. The file-I/O overhead from chapter 11 is the price, and for processing untrusted content the secure-deployment guide's own judgment is that it is usually worth it. Measure your workload against that 10-to-200x file-I/O range before committing, but the isolation is the right shape.

A regulated environment handling PII

An agent operates on personal data under a compliance regime, likely multi-tenant, where a cross-tenant leak is a reportable breach, not merely an internal incident. Here the top branches of the tree apply together: regulated data plus multiple tenants, and often an explicit requirement that workloads not share a kernel. That points to a microVM, the hardware boundary from chapter 12, with its separate guest kernel, five-device attack surface, and the jailer as a second line of defense. Most teams will consume this through a managed service rather than run Firecracker themselves, which is what chapter 19 builds on AWS Lambda MicroVMs. The point of this scenario is that the strongest boundary is chosen because the consequences of failure, not the technology's appeal, demand it.

🏭 In production

These layers combine, and the best deployments stack them. A multi-tenant service might run each session in a microVM, harden the container inside it anyway, and still route all egress through the proxy Part IV builds. Defense in depth means a single failure, a kernel bug, a misconfigured capability, a too-broad allowlist, does not by itself hand over the system. The decision tree picks your primary boundary. It does not stop you adding the cheaper controls underneath it, and you generally should.

🔬 Under the hood

Notice what none of these boundaries do on their own: control what the agent reaches once it is allowed to make a network request. Every scenario above ended with traffic going through a proxy, because the isolation boundary and the network boundary are different controls answering different questions. Isolation decides what the code can touch on the machine. The proxy decides where its requests can go and injects the credentials it should never hold. Part IV is that second boundary, and it applies no matter which rung of this chapter's ladder you landed on.
