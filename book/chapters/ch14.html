Chapter 14 · Part IV — Credentials & the Network Boundary

The Proxy Pattern

An agent needs credentials to be useful: a GitHub token to push, an API key to call a service, cloud keys to read a bucket. The moment one of those secrets sits in the agent's environment, it is one prompt injection away from being read back to an attacker. This chapter is about the move that takes the secret off the table entirely. You put a proxy between the agent and the network, the proxy holds the credential, and the agent sends requests that carry none. A credential the agent never holds is a credential it can never leak.

The security boundary, applied to a secret

Chapter 3 introduced the idea of a security boundary: a line between components that trust each other differently, drawn so that a failure on one side does not become a failure on the other. Credentials are where that idea pays off most directly. The Agent SDK guide frames it as putting sensitive resources outside the boundary that contains the agent, so that if something goes wrong in the agent's environment, the resources outside stay protected.

The secure-deployment guide gives the canonical example: rather than handing an agent an API key, run a proxy outside the agent's environment that injects the key into requests. The agent can make the API call. It never sees the key. That is the whole pattern, and the rest of this chapter is about the mechanics of making an agent's traffic actually go through such a proxy, and the two rather different ways to handle Anthropic's own API versus every other service.

What the proxy buys you

A proxy that sits on the credential is doing four jobs at once, and each one is a control you would otherwise have to build separately. The agent never sees the actual credential, so a leak of the agent's environment leaks nothing useful. The proxy enforces an allowlist of permitted endpoints, so a request aimed at an attacker's server never leaves. Every request passes through one place that can log it for audit. And the secret lives in exactly one location you can rotate, rather than being copied into every agent's environment. The guide lists these as the benefits of the pattern, and they compound: the same box that hides the credential also decides where traffic is allowed to go.

🔬 Under the hood

The hardened container from chapter 10 runs with --network none and reaches the outside world only through a mounted Unix socket wired to a proxy on the host. The proxy pattern is what makes that topology useful: with no network interface of its own, the agent literally cannot address a server the proxy will not forward to. Injecting a credential at that same proxy is a small addition to a boundary you have already drawn.

Pointing Claude Code at a proxy

There are two ways to route the agent's traffic, and they are not interchangeable. The first is specific to Anthropic's sampling API and gives your proxy plaintext to work with. The second is the standard system-wide proxy mechanism and, for HTTPS, hands your proxy an opaque tunnel it cannot read without extra work.

ANTHROPIC_BASE_URL tells Claude Code and the Agent SDK to send sampling requests somewhere other than the Claude API directly. Your proxy receives plaintext HTTP, so it can inspect the request, inject the real API key, and forward it on. This is the simple path, and the reason it is simple is that it only covers requests to the model.

HTTP_PROXY and HTTPS_PROXY are the standard environment variables, and Claude Code respects them for all HTTP traffic. The catch is TLS. For an HTTPS destination the client opens an encrypted CONNECT tunnel through the proxy, and, as the guide puts it, the proxy cannot see or modify the request contents without TLS interception. It can see the hostname and decide whether to allow the connection. It cannot read or change what flows inside.

Credentials for services that are not the Claude API

Sampling is the easy case because ANTHROPIC_BASE_URL gives the proxy plaintext. Every other authenticated service the agent touches, a git remote, a package registry, an internal API, speaks TLS end to end, and routing it through HTTPS_PROXY only gets the proxy an opaque tunnel. The guide describes two ways out, and they trade off cleanly against each other.

The first is a custom tool. You expose the capability through an MCP server or custom tool whose implementation runs outside the agent's boundary and makes the authenticated request there. The agent calls the tool; the credential is added on the far side of the boundary; the agent only ever sees the tool interface. A git MCP server that forwards commands to a git proxy on the host is the guide's example. The appeal is that nothing has to decrypt anyone's TLS: the service outside the boundary makes its own authenticated connection, and the credential simply never crosses into the agent.

The second is traffic forwarding, and it is the heavier option. To modify arbitrary HTTPS without writing a custom tool, you run a TLS-terminating proxy that decrypts the traffic, inspects or rewrites it, then re-encrypts it before forwarding. The guide spells out the cost: you run the proxy outside the container, you install the proxy's CA certificate in the agent's trust store so the agent trusts the certificates the proxy presents, and you point HTTP_PROXY/HTTPS_PROXY at it. This handles any HTTP-based service without per-service code, at the price of managing a certificate authority and decrypting traffic that was encrypted for a reason. Chapter 15 builds exactly this, with Squid, and does not gloss over what decrypting someone's TLS actually means.

⚠ Gotcha

Setting HTTP_PROXY/HTTPS_PROXY does not guarantee a program honors them. curl, pip, npm, and git do, but some runtimes connect directly. Node.js fetch() ignores these variables by default; on Node 24 and later you enable support with NODE_USE_ENV_PROXY=1. For coverage that does not depend on each tool cooperating, the guide points to proxychains to intercept network calls, or iptables to redirect outbound traffic into a transparent proxy the client never had to be told about.

Which proxy

The guide names four proxies you can build on rather than writing your own. Envoy is a production-grade proxy whose credential_injector filter adds auth headers. mitmproxy is a TLS-terminating proxy built for inspecting and modifying HTTPS. Squid is a caching proxy with a mature access-control-list system. LiteLLM is an LLM gateway with credential injection and rate limiting. Any of them can hold a credential the agent does not. The next chapter builds the Squid path start to finish, because its ACL model maps directly onto the allowlist-then-inject shape this pattern wants, and because it runs the same way in your lab as it will in the ECS sidecar of chapter 18.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — the repo and Python environment are already set up.

Run python3 examples/ch14/inject_demo.py. It stands up a mock upstream and an injecting proxy on localhost, sends one request through the proxy with no Authorization header, and prints what the upstream received. The client sends nothing; the upstream sees the injected credential. That is the entire pattern in stdlib, before Squid makes it real in the next chapter.

Then open challenges/ch14/ and implement proxy_pattern_prevents so the tests pass: uv run pytest challenges/ch14. You are deciding, for a set of exposure scenarios, whether the two mechanisms of the pattern, removing the secret from the agent and blocking egress, actually stop the leak, and finding the one case where they do not.
